## 13.3.2 向量接口（续）

### 优化示例代码（续）

下面的 `main` 函数展示了在 Global 内存中分配空间、Host 与 Device 间数据拷贝、kernel 调用与释放的完整流程：

```c
{
    // 设置使用0号SPA
    sdaaSetDevice(0);
    int h_glb_data[16] = { 0 };

    for (int i = 0; i < 16; i++){
        h_glb_data[i] = i;
    }

    int *glb_data = NULL;
    // 在Global内存中malloc一段空间
    sdaaMalloc((void **)&glb_data, 16 * sizeof(int));
    // 将数据从Host端拷贝到Device端
    sdaaMemcpy(glb_data, h_glb_data, 16 * sizeof(int), sdaaMemcpyHostToDevice);

    // kernel函数调用
    test_func<<<1>>>(glb_data);
    // 等待kernel函数执行完成
    sdaaDeviceSynchronize();

    // 释放申请的Global内存
    sdaaFree(glb_data);
    return 0;
}
```

### 优化效果

| 接口名称 | 接口性能（执行周期，单位：拍）（源地址和目的地址的数据均位于 Global 存储空间） | 接口性能（执行周期，单位：拍）（源地址和目的地址的数据均位于 SPM 存储空间） |
|---|---|---|
| simd_load | 758 | 12 |

在本示例中，源地址和目的地址的数据均位于 SPM 存储空间，比源地址和目的地址的数据均位于 Global 存储空间，向量接口性能快约 63 倍。

### 13.3.3.2 优化策略

SDAA C 提供的数学函数接口，除标量（单精度、双精度）外还提供了向量接口和 128 接口：

| 接口类别 | 输入类型 | 输出类型 | 说明 |
|---|---|---|---|
| 单精度标量接口 | 单精度浮点数 | 单精度浮点数 | 与标准库一致，计算一个单精度浮点数的数学函数值。 |
| 双精度标量接口 | 双精度浮点数 | 双精度浮点数 | 与标准库一致，计算一个双精度浮点数的数学函数值。 |
| 向量接口 | floatv16 向量类型 | floatv16 向量类型 | 利用 SIMD 指令，一次计算 16 个单精度浮点数的数学函数值。 |
| 128 接口 | float 型指针 | 无 | 利用 SIMD 指令和循环展开，一次计算 128 个单精度浮点数的数学函数值，并返回覆盖原地址上数据。 |

#### 优化策略

需要计算大量连续的单精度浮点数时，采用向量接口或者 128 接口相比使用单精度标量接口计算每个元素，性能更好。

#### 优化示例

以指数函数 exp 为例，使用一个 SPE 对 51200 个单精度浮点数进行计算，测试并比较以下三种接口的执行周期：

- 单精度标量接口 expf。
- 向量接口 simd_exp。
- 128 接口 simd_exp128。

##### 测试环境

| TecoDriver 和 TecoToolKit 版本 | 计算核心频率 |
|---|---|
| v1.2.0 | 2.36GHz |

##### 测试代码

```c
#include <sdaa_perf.h>
using namespace sdaa;

#define NUM 51200
#define MIN -80
#define MAX 80

__device__ void init_arr(float *arr)
{
        for (int i = 0; i < NUM; i++) {
            arr[i] = ((float)(MAX - MIN) / (float)NUM) * i + MIN;
        }
}

__global__ void test_expf()
{
        if (threadIdx != 0) return;
        // 在SPM内存中申请一段空间
        float *arr = (float *)malloc(NUM * sizeof(float));
        init_arr(arr);
        // 收集性能数据起始值
        perf_start();
        for (int i = 0; i < NUM; i++) {
            arr[i] = expf(arr[i]);
        }
        // 收集性能数据结束值
        perf_stop();
        // 输出性能数据
        perf_print();
        // 释放申请的SPM内存空间
        free(arr);
}

__global__ void test_simd_exp()
{
    if (threadIdx != 0) return;
    // 在SPM内存中申请一段空间
    float *arr = (float *)malloc(NUM * sizeof(float));
    init_arr(arr);
    // 收集性能数据起始值
    perf_start();
    for (int i = 0; i < NUM; i += 16) {
        floatv16 input;
        simd_load(input, arr + i);
        input = simd_exp(input);
        simd_store(input, arr + i);
    }
    // 收集性能数据结束值
    perf_stop();
    // 输出性能数据
    perf_print();
    // 释放申请的SPM内存空间
    free(arr);
}

__global__ void test_simd_exp128()
{
    if (threadIdx != 0) return;
    // 在SPM内存中申请一段空间
    float *arr = (float *)malloc(NUM * sizeof(float));
    init_arr(arr);
    // 收集性能数据起始值
    perf_start();
    for (int i = 0; i < NUM; i += 128) {
        simd_exp128(arr + i);
    }
    // 收集性能数据结束值
    perf_stop();
    // 输出性能数据
    perf_print();
    // 释放申请的SPM内存空间
    free(arr);
}

int main()
{
    // 设置使用2号SPA
    sdaaSetDevice(2);
    // kernel函数调用
    test_expf<<<1>>>();
    // 等待kernel函数执行完成
    sdaaDeviceSynchronize();
    // kernel函数调用
    test_simd_exp<<<1>>>();
    // 等待kernel函数执行完成
    sdaaDeviceSynchronize();
    // kernel函数调用
    test_simd_exp128<<<1>>>();
    // 等待kernel函数执行完成
    sdaaDeviceSynchronize();

    return 0;
}
```

##### 优化效果

| 接口名称 | 51200 次数学函数计算的指令周期（拍） | 平均每次数学函数计算的指令周期（拍） |
|---|---|---|
| 单精度标量接口 expf | 7885552 | 154.02 |
| 向量接口 simd_exp | 650860 | 12.71 |
| 128 接口 simd_exp128 | 354080 | 6.92 |

在本示例中，使用向量接口的性能是标量接口的约 12.11 倍，使用 128 接口的性能是标量接口的约 22.26 倍。

## 13.3.4 程序编译

### 13.3.4.1 循环展开

本文主要介绍使用循环展开的方式提高性能的优化策略和示例。

#### 原理介绍

循环展开是在编译程序中，通过减少循环迭代次数进而提高程序执行速度的方法。

其核心思想是将操作对象线性化，在一次迭代中访问线性数据中的一组而非单个，使程序在执行更少的迭代次数后，能够有效降低循环的开销。同时，循环展开可以减少指令的相关性，增加循环体内语句并发执行的可能性，有助于进行更好的指令调度；使用循环展开还可以减少分支预测失败的可能性。但循环展开也会使代码膨胀；代码的可读性降低，以及寄存器压力增大。

循环展开可以在程序编写时完成，也可以由编译器自动完成。由于编译器可以更好地了解目标体系结构的工作原理，并能够在有益的情况下展开循环。因此，建议您使用编译器自动展开，如果您手动展开，建议您在测试手动循环展开确实有效再进行。

#### 优化策略

通过以下编译指示，控制是否需启用循环展开功能，以及展开的次数：

- `#pragma clang loop unroll(enable)`：启用基于自动诊断的展开功能。
- `#pragma clang loop unroll(disable)`：禁用基于自动诊断的展开功能。
- `#pragma clang loop unroll(full)`：完全展开一个循环。
- `#pragma clang loop unroll_count(N)`：显式指定展开 N 次，其中 N 是编译时的常量。

#### 优化示例

使用不同的编译指示，在 `for` 循环内对 1000 个整型数进行“+1”的操作，测试以下情况的执行周期并比较：

- 不做任何操作
- `#pragma clang loop unroll(enable)`：启用基于自动诊断的展开功能。
- `#pragma clang loop unroll(disable)`：禁用基于自动诊断的展开功能。
- `#pragma clang loop unroll(full)`：完全展开一个循环。
- `#pragma clang loop unroll_count(1)`：显式指定展开 1 次。
- `#pragma clang loop unroll_count(2)`：显式指定展开 2 次。
- `#pragma clang loop unroll_count(3)`：显式指定展开 3 次。
- `#pragma clang loop unroll_count(4)`：显式指定展开 4 次。
- `#pragma clang loop unroll_count(5)`：显式指定展开 5 次。
- `#pragma clang loop unroll_count(6)`：显式指定展开 6 次。
- `#pragma clang loop unroll_count(7)`：显式指定展开 7 次。
- `#pragma clang loop unroll_count(8)`：显式指定展开 8 次。
- `#pragma clang loop unroll_count(9)`：显式指定展开 9 次。
- `#pragma clang loop unroll_count(10)`：显式指定展开 10 次。

每个测例均调用 10000 次“+1”的操作计算平均值以消除初次调用时 cache 影响。

| TecoDriver 和 TecoToolKit 版本 | 计算核心频率 |

| v1.2.0 | 2.36GHz |

##### 测试代码

```c
#include <sdaa_perf.h>
using namespace sdaa;

#define N 10000
#define SIZE 1000

__device__ void init_data(int *data)
{
    for (int i = 0; i < SIZE; i++) {
        data[i] = i;
    }
}

__device__ void check_result(int *data)
{
    for (int i = 0; i < SIZE; i++) {
        if (data[i] != i + N) {
              printf("calculate error\n");
        }
    }
}

__device__ void loop_unroll_do_nothing()
{
    int data[SIZE];
    init_data(data);

    // 收集性能数据起始值
    perf_start();
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < SIZE; j++) {
              data[j] = data[j] + 1;
        }
    }
    // 收集性能数据结束值
    perf_stop();

    check_result(data);
    return;
}

__device__ void loop_unroll_enable()
{
    int data[SIZE];
    init_data(data);

    // 收集性能数据起始值
    perf_start();
    for (int i = 0; i < N; i++) {
        #pragma clang loop unroll(enable)
        for (int j = 0; j < SIZE; j++) {
              data[j] = data[j] + 1;
        }
    }
    // 收集性能数据结束值
    perf_stop();

    check_result(data);
    return;
}

__device__ void loop_unroll_disable()
{
    int data[SIZE];
    init_data(data);

    // 收集性能数据起始值
    perf_start();
    for (int i = 0; i < N; i++) {
        #pragma clang loop unroll(disable)
        for (int j = 0; j < SIZE; j++) {
              data[j] = data[j] + 1;
        }
    }
    // 收集性能数据结束值
    perf_stop();

    check_result(data);
    return;
}

__device__ void loop_unroll_full()
{
    int data[SIZE];
    init_data(data);

    // 收集性能数据起始值
```

### 13.3.4.2 链接时优化（LTO）

本文主要介绍使用链接时优化（LTO）提高性能的优化策略和示例。

#### 原理介绍

链接时优化（LTO）是对整个 SDAA C 程序在链接时进行的跨模块间优化，聚焦于分析整个程序代码，属于过程间优化（IPO）的一种。

具体实现原理是将一些 pass 的优化推迟到链接时再进行，从而使得这些 pass 具有可跨模块优化的能力。

LTO 可以实现：

- **更好的内联**：LTO 可以在链接阶段分析整个程序的调用关系，从而实现智能的函数内联决策，提高程序性能。
- **消除死代码**：LTO 可以分析整个程序的数据流和控制流，检测并删除不会被执行的代码，从而减小程序大小。
- **传播跨模块的常量**：LTO 可以在整个程序范围内传播已知的常量值，从而提高程序性能。
- **消除跨模块的重复代码**：LTO 可以在整个程序范围内检测并消除重复的代码片段，从而减小程序大小。

#### 优化策略

通过以下编译选项，控制是否需启用 LTO 优化：

- `-flto`：启用 LTO 优化，编译和链接时使用。
- `--sdaa-link`：启用 LTO 优化，链接时使用，指示链接 SDAA C 的 `clang-offload-bundler` 捆绑包。

#### 优化示例

分别使用以下两种方式编译示例程序，测试执行周期并比较：

- 不使用 LTO 优化
- 使用 LTO 优化

每个测试均调用 10000 次 `foo1` 函数并计算平均值以消除初次调用时 cache 影响。

##### 不使用 LTO 优化

执行以下编译命令，将 `lib.scpp` 编译成静态库，并使用 `main.scpp` 链接该静态库生成可执行文件：

```bash
# 将lib.scpp编译生成.o
tecocc lib.scpp --sdaa-device-only -c -o nonlto.o
# 将生成的.o打包成静态库
ar -rcs nonlto.a nonlto.o
# 编译main.scpp并链接静态库nonlto.a生成可执行文件
tecocc main.scpp --sdaa-static-device-lib=nonlto.a -o nonlto.out
```

##### 使用 LTO 优化

执行以下编译命令，将 `lib.scpp` 编译成静态库，并使用 `main.scpp` 链接该静态库生成可执行文件：

```bash
# 将lib.scpp编译生成.o
tecocc lib.scpp --sdaa-device-only -flto -c -o lto.o
# 将生成的.o打包成静态库
ar -rcs lto.a lto.o
# 编译main.scpp并链接静态库lto.a生成可执行文件
tecocc main.scpp --sdaa-static-device-lib=lto.a -flto --sdaa-link -o lto.out
```

### 13.3.4.3 浮点融合

本文主要介绍使用浮点融合提高性能的优化策略和示例。

#### 原理介绍

浮点融合是使用一条乘加/乘减运算操作代替乘法和加/减法两条运算的操作，从而提高程序性能并减小误差。

#### 优化策略

通过以下编译选项控制是否需启用浮点融合功能：

- `-ffp-contract=fast`：启用浮点融合，忽略 `#pragma FP_CONTRACT`。
- `-ffp-contract=on`：根据 `#pragma FP_CONTRACT` 决定是否启用浮点融合。
- `-ffp-contract=off`：禁用浮点融合，忽略 `#pragma FP_CONTRACT`。

默认为启用浮点融合，您可以调整浮点运算写法，拼凑尽量多的乘加类操作，帮助编译器优化。

目前支持以下几种操作的融合：

- 单精度乘加（单精度同号相乘再做加法）
- 双精度乘加（双精度同号相乘再做加法）
- 单精度乘减（单精度同号相乘再做减法）
- 双精度乘减（双精度同号相乘再做减法）
- 单精度负乘加（单精度异号相乘再做加法）
- 双精度负乘加（双精度异号相乘再做加法）
- 单精度负乘减（单精度异号相乘再做减法）
- 双精度负乘减（双精度异号相乘再做减法）

#### 优化示例

使用不同的编译选项，在 `for` 循环内进行 100 次乘法和加/减操作，测试以下情况的执行周期并比较：

- 单精度负乘加
- 双精度负乘加
- 单精度负乘减

每个测例均调用 10000 次乘法和加/减操作计算平均值以消除初次调用时 cache 影响。
