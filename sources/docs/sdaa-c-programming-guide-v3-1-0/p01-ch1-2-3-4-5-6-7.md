# SDAA C 编程指南 v3.1.0（第1部分）

## 1. 概述

CUDA C/CUDA Fortran、OpenCL 等编程语言在编程语法上符合当前主流编程语言的编程规范，学习成本较低。但是其对应的编程模型与硬件结合较为密切，需要用户深入了解硬件体系结构，否则可能无法有效使用加速核心的全部算力。OpenACC 屏蔽了底层的硬件差异，不需要深入了解硬件体系结构，但是其自身的编程导语与主流的编程语言存在较大差异，需要用户全量学习 OpenACC 的编程语法，学习成本和使用门槛均较高。

为了解决主流众核加速产品在学习成本和使用方法上的问题，SDAA C 编程语言应运而生。

SDAA C（Software Defined Accelerator Architecture C）是运行在 SDAA 异构并行计算平台上的编程模型。以太初 AI 加速卡为计算核心，支持 C/C++ 编程语言，用户仅需学习少量的 SDAA C 编程语法即可快速上手编程。同时 SDAA C 最大化地隐藏了底层硬件的实现细节，用户无需深入了解硬件的体系架构就可以正常使用太初 AI 加速卡提供的强大算力，大幅度地降低学习成本，使用户可以专注于并行程序的开发。

### 1.1 SDAA 异构并行计算平台

SDAA 异构并行计算平台是太初元碁针对 AI 场景，自主研发的众核异构计算架构，能够充分发挥太初 AI 加速卡的计算能力，为科学计算、人工智能等领域提供完整的解决方案。

> 图1 SDAA 异构并行计算平台

SDAA 异构并行计算平台以太初元碁人工智能系列硬件（太初 AI 加速卡）为基础，基础软件层完成对硬件驱动及操作系统的高度抽象，并提供完整的运行时库，编程模型及配套的编译、调试和分析工具；加速库提供包括 TecoDNN、TecoBLAS 等基础且丰富的算子库；应用层整合了当前主流的深度学习框架。

SDAA C 运行于基础软件层，对下屏蔽了硬件的物理特性，对上提供了丰富的编程接口，以及代码调试、调优工具。用户无需感知底层的硬件细节，仅需遵从 SDAA C 的编程规范即可使用太初 AI 加速卡，为高并发、大数据运算提供强有力的算力支撑。

### 1.2 SDAA 异构并行模型

SDAA 异构并行模型采用主从异构的物理架构，即 Host-Device 运行模式。

> 图2 SDAA 异构并行模型

- **主机端（Host 端）**：由 CPU 构成，主要负责对设备端（Device 端）内存资源的申请和释放，并控制设备端（Device 端）处理任务的资源。
- **设备端（Device 端）**：由太初 AI 加速卡构成，负责处理大规模的数据运算。太初 AI 加速卡通过高效的运算部件（SPE）和访存部件（Global 存储器等）弥补了通用处理器在处理高并发、大数据场景下计算和访存上的短板，提高了数据处理的效率。
  - 太初 AI 加速卡由计算核心阵列 SPA（Synergistic Processor Element Array）和对应的 Global 高带宽存储器构成。
  - 每个计算核心阵列 SPA 是一个协从运算单元，由计算核心 SPE（Synergistic Processor Element）构成。

> 图3 太初 AI 加速卡

### 1.3 产品优势

SDAA C 规避了主流加速器编程模型中存在的编程语法学习成本高、与硬件强相关造成的兼容性差等诸多缺点，是一款上手简单，同时内容丰富的编程模型。

**学习成本低**

SDAA C 支持 C/C++ 标准语法，用户仅需学习少量的 SDAA C 编程语法即可快速上手编程。同时 SDAA C 最大化地隐藏了底层硬件的实现细节，用户无需深入了解硬件的体系架构就可以正常使用太初 AI 加速卡提供的强大算力，大幅度地降低学习成本，使用户可以专注于并行程序的开发。

**丰富的函数接口**

SDAA C 提供了包括：矩阵乘、向量运算函数、数据搬运及原子操作等丰富的函数接口，可以更好地辅助用户使用太初 AI 加速卡的计算资源，简化应用程序开发，优化应用程序性能。

**丰富的数学函数**

SDAA C 内置了丰富的数学函数接口，集成了包括：正弦计算、余弦计算、正切计算、指数计算、对数计算、高斯误差计算等常用标量数学计算接口。同时为了满足用户对于应用程序运行性能的追求，还提供了相应的向量数学函数计算接口，具体可参考数学函数。

**程序调试**

程序调试是应用程序开发过程中的必要环节，优秀的程序调试手段可以帮助程序开发人员在一定程度上推进应用程序开发的进度。SDAA C 提供打印计算核心 SPE 接口运行时日志功能，可以帮助用户快速定位设备端的运行时问题，具体可参考程序调试。

**性能调优**

为了获取更好的系统性能并满足不断增加的业务需求，程序的运行性能通常是应用程序开发人员重点关注的内容之一，SDAA C 提供了性能调优工具和基本的并行编程优化技巧，可以辅助程序开发人员优化应用程序，最大化的利用太初 AI 加速卡的物理资源，提升应用程序的执行速度，具体可参考性能调优。

### 1.4 工作流程

SDAA C 将 Host 端的应用程序切分成一系列的加速任务，每个加速任务会通过 SDAA C 的编程接口发送到 Device 端的太初 AI 加速卡上并行执行。加速任务在 Device 端执行完毕后会将计算结果返回至 Host 端，供用户进一步处理。

> 图1 SDAA C 工作流程

### 1.5 设备端代码使用限制

SDAA C 基于 C/C++ 语法，并在此基础上进行了部分优化扩展。用户在设备端编程时，可以基本遵循 C/C++ 编程规范。SDAA C 的设备端代码不支持以下内容：

- Exceptions and try/catch blocks 异常处理
- RTTI
- STL 标准库
- 全局对象的构造与析构
- 虚函数表跨越 Host 与 Device 端
- 使用 new 关键字
- `__global__` 修饰的函数中使用可变参数
- 线程私有操作
- 局部静态变量
- C/C++ 原生的原子操作（您可以使用 SDAA C 提供的原子操作接口，可参考原子操作）
- 以下终止进程相关函数：

```c
void exit(int status);
int atexit(void (*function)(void));
```

- 所有文件操作相关函数：

```c
int fclose(FILE *stream);
FILE *fopen(const char *pathname, const char *mode);
FILE *fdopen(int fd, const char *mode);
FILE *freopen(const char *pathname, const char *mode, FILE *stream);
```

## 4. 异构计算模型

### 4.1 概述

CPU 处理器体系偏向于快速的用户反馈和调度控制，但其计算能力有限。在面向高并发、大规模的数据计算时，需要使用异构计算模型对计算进行加速，从而节省用户大批量数据计算所消耗的时间。使用 SDAA C 提供的异构计算模型，用户无需感知底层的硬件细节，即可正常使用太初 AI 加速卡提供的强大算力。

SDAA C 异构计算模型是将应用程序中大量的计算任务，由主机端发送到设备端的计算核心阵列 SPA 执行，充分利用太初 AI 加速卡的硬件资源，提高程序的运行速度，缩短程序的运行时间。

> 图1 异构计算模型

### 4.2 SPMD

SPMD（Single Program Multiple Data，单程序多数据），是一种用于任务并行的编程范式。其本质是将一个问题分解成若干个子问题，然后对其并行求解。

SDAA C 遵循 SPMD 编程范式，同一计算核心阵列 SPA 内所有计算核心 SPE 运行同一份应用程序，每个 SPE 可以独立完成对子问题的并行求解。其核心在于对复杂计算任务的切分，并合理地将计算任务分配到不同的 SPE 内进行计算。

> 图1 基于 SDAA C 的 SPMD 编程范式

SDAA C 提供了 threadIdx 和 threadDim 为每个 SPE 定制计算任务。其中：

- `threadIdx`：当前计算核心 SPE 的 ID 号，可以通过该关键字为目标 SPE 分配计算任务。
- `threadDim`：获取同一计算核心阵列 SPA 内计算核心 SPE 的总数。

**编程示例**

以数组计算为例，将计算任务分配到同一 SPA 的所有 SPE 内并行执行：

```c
using namespace sdaa;
#define SIZE 67


__device__ int g_data[SIZE];
__global__ void func()
{
     // 初始化原始数据
     for (int i = 0; i < SIZE; i++) {
         g_data[i] = i;
     }


     // SPE同步，确保所有的SPE都完成了数据的初始化
     sync_threads();


     // 可用的计算资源超过计算量时，每个SPE承担一次计算
     if (threadDim >= SIZE) {
         if (threadIdx < SIZE) {
             g_data[threadIdx]++;
         }
     } else {
         // 可用计算资源少于计算量时，每个SPE需承担多次计算
         // 例如当前计算量为67，假如共有32个SPE参与计算。则每个SPE需要至少承担两次计算任务
         int cal_time = SIZE / threadDim;
         for (int i = 0; i < cal_time; i++) {
             g_data[i * threadDim + threadIdx]++;
         }


         // 对于剩余的计算量使用部分SPE承担
         // 例如当前的计算量为67，共有32个SPE参与计算，每个SPE计算两次后，还剩余3次。则使用0， 1， 2三个SPE完成剩余的计算任务
         int remain_time = SIZE % threadDim;
         if ((threadIdx + 1) <= remain_time) {
             g_data[cal_time * threadDim + threadIdx]++;
         }
     }


     // SPE同步，确保所有的SPE都完成了子问题的求解
     sync_threads();


     // 打印输出计算结果
     for (int i = 0; i < SIZE; i++) {
         printf("SPE ID = %lu, g_data[%d] = %d\n", threadIdx, i, g_data[i]);
     }
}
```

上述应用程序中就通过 SPMD 编程范式完成对复杂任务的求解。

- **SP（Single Program，单程序）**：所有参与计算的 SPE 都执行同一份代码，在本例中为 func 函数。
- **MD（Multiple Data，多数据）**：通过 `threadIdx` 将数组进行切分，每个 SPE 仅需完成少量运算。
- 在本例中：SIZE 为 67，假如可用计算资源为 32 个 SPE，计算资源少于计算量，每个 SPE 需要承担多次计算，如：SPE0 需要计算数据 g_data[0]、g_data[32] 和 g_data[64]、SPE1 需要计算数据 g_data[1]、g_data[33] 和 g_data[65] 等。

> 图2 使用 SPMD 编程范式完成数组计算任务

具体计算任务的代码片段如下：

```c
// 可用计算资源少于计算量时，每个SPE需承担多次计算
// 例如当前计算量为67，共有32个SPE参与计算。则每个SPE需要至少承担两次计算任务
int cal_time = SIZE / threadDim;
for (int i = 0; i < cal_time; i++) {
    g_data[i * threadDim + threadIdx]++;
}


// 对于剩余的计算量使用部分SPE承担
// 例如当前的计算量为67，共有32个SPE参与计算，每个SPE计算两次后，还剩余3次。则使用0， 1， 2三个SPE完成剩余的计算任务
int remain_time = SIZE % threadDim;
if ((threadIdx + 1) <= remain_time) {
    g_data[cal_time * threadDim + threadIdx]++;
}
```

代码中涉及部分关键字含义如下：

- `__device__`：运行空间标识符，修饰 Device 端的函数或变量，在 Device 端调用、运行，可参考 `__device__`。
- `__global__`：函数标识符，修饰 Kernel 函数，可参考 `__global__`。
- `sync_threads`：对同一计算核心阵列 SPA 内的所有或部分计算核心 SPE 进行同步操作，可参考 `sync_threads`。
- `printf`：打印接口，可参考 `printf`。

**SPMD 编程注意事项**

考虑到程序的可扩展性，在为 SPE 分配计算任务时，应尽量考虑到以下几点：

- 负载均衡，每个 SPE 承担的计算量尽可能保持一致。
- 数据就近原则，SPE 内用于计算的原始数据尽可能来自于当前 SPE，减少同一计算核心阵列 SPA 内不同计算核心 SPE 之间的数据传递。
- 分配计算任务时，请提前做好规划，以便在计算量增加时可以做到更好地兼容与扩展。

### 4.3 Kernel 函数

太初 AI 加速卡负责处理大规模的数据运算（可参考 SDAA 异构并行模型），计算核心 SPE 在 SDAA C 中相当于一个线程，用户可以使用 SPE 进行异构并行计算。

Kernel 函数是由 `__global__` 关键字修饰的函数，在设备端执行（类似于线程代码）。在使用 SDAA C 时，主机端通过调用 Kernel 函数将大量的计算任务分配给 SPE 执行，从而提高数据的处理效率。

> 图1 Kernel 函数

SDAA C 编程语言：

- 通过 `__global__` 关键字标明 Kernel 函数，可参考 `__global__` 关键字。
- 在主程序中使用 `<<<…>>>` 语法调用 Kernel 函数，将计算任务发送到太初 AI 加速卡上执行计算，可参考 `<<<...>>>` 语法。
- 使用 `threadIdx` 标识执行计算的 SPE，可参考 `threadIdx` 关键字。

以执行加法为例的 Kernel 函数：

```c
// Kernel函数
__global__ void user_kernel(int *a)
{
    int tid = threadIdx;
    // 每个SPE执行一个加法操作，threadDim个SPE完成threadDim个加法操作
    a[tid] = a[tid] + a[tid];
    return;
}
```

### 4.4 存储空间

SDAA C 的存储空间分为主机端内存、设备端内存、计算核心 SPE 片上存储空间 SPM。

> 图1 存储模型

**主机端内存**

主机端内存是主机端的存储空间，用于存储需要在设备端进行计算的源数据。

申请和释放主机端内存：可以使用 `malloc`、`free` 函数，可参考 C/C++ 语言的相关内容。

**设备端内存（Global 存储空间）**

设备端内存又称设备端 Global 存储空间，是计算核心阵列 SPA 的共享存储。在使用计算核心 SPE 进行计算前，需要先将数据从主机端传输到设备端的 Global 存储空间。

Global 存储空间可以使用以下方法进行管理：

- 使用 `__device__` 关键字申请 Global 存储空间。
- 使用 `sdaaMalloc`、`sdaaFree` 申请和释放 Global 存储空间。

Global 存储空间使用示例：

- 使用 `__device__` 关键字分配 Global 存储空间。

```c
// g_mem被创建在global存储空间内
__device__ int g_mem = 1;
```

- 使用 `sdaaMalloc` 申请 Global 存储空间。
- 使用 `sdaaFree` 释放申请的 Global 存储空间。

```c
// 需要在Host端使用sdaaMalloc申请Global存储空间，使用sdaaFree释放申请的Global存储空间
#define size 100;
int main()
{
    int *h_mem = NULL;
    // 为h_mem申请Global存储空间
    sdaaMalloc((void **)(&h_mem), size * sizeof(int));


    ...


    // 释放h_mem
    sdaaFree(h_mem);
}
```

**SPM 存储空间**

每个计算核心 SPE 都有一块高速的片上本地局部数据存储空间 SPM。SPM 空间属于计算核心 SPE 的私有空间。对于计算核心 SPE 而言，访问 Global 存储空间，速度缓慢。在 SPE 执行计算任务的过程中，如果直接从 SPM 存储空间获取数据，计算效率将得到显著提升。

计算核心 SPE 运行时独占 SPM 存储空间，SPM 存储空间可分为：

- **堆空间**：需要手动管理的一块动态内存空间，您可以通过 malloc 接口分配堆内存空间，通过 free 接口释放所申请的堆内存空间。可通过 get_heap_size 接口查询当前计算核心 SPE 的堆空间总大小。
- **栈空间**：程序运行时用于管理函数调用和局部变量的内存空间。可通过 get_stack_size 接口查询当前计算核心 SPE 的栈空间总大小。
- **local 空间**：使用 `__local__` 关键字声明的空间，不支持变量初始化，可通过 get_local_size 接口查询当前计算核心 SPE 的 local 空间总大小。

> **说明：**
>
> 在开启 `--stack-on-global` 选项后，`__local__` 和 `__scoped_local__` 共享 local 空间。

SPM 存储空间使用示例：

```c
// 使用__local__关键字分配local空间
__local__ int a;


__device__ void user_spm()
{
    // 变量b被分配到了user_spm函数的栈空间内
    int b = 1;


    // 使用malloc接口动态分配堆空间
    int *c = (int *)malloc(16 * sizeof(int));


    // 使用free接口释放所申请的堆空间
    free(c);
}
```

**存储空间可用大小**

| 存储空间名称 | 存储空间大小 | 所属存储空间 |
|---|---|---|
| 堆空间 | 通过 get_heap_size 接口查询 | SPM 存储空间 |
| 栈空间 | 通过 get_stack_size 接口查询 | 默认为 SPM 存储空间；在开启 `--stack-on-global` 选项后，切换到 Global 存储空间 |
| local 空间 | 通过 get_local_size 接口查询 | SPM 存储空间 |
| Global 存储空间 | 16GB | Global 存储空间 |

### 4.5 硬件架构

太初 AI 加速卡由多个计算核心阵列 SPA 组成，每个计算核心阵列 SPA 由 threadDim 个计算核心 SPE 和 Global 存储空间构成。每个计算核心 SPE 都拥有独立的计算、访存和控制功能，是一个独立的处理器核心，可以独立完成一个计算任务。每个计算核心 SPE 由 FU、SU、SREG、VPU、VREG 以及 SPM 存储空间组成。

> 图1 太初 AI 加速卡

**SU**

SU（Scalar Unit）标量计算单元，用于执行标量运算的部件，可以完成算术运算、逻辑运算等标量运算。标量计算单元只能接收标量寄存器内的数据，在完成运算后将计算结果保存到标量寄存器内。

**SREG**

SREG（Scalar Register）标量寄存器，用于存储标量数据的寄存器，支持单个数据元素的读写操作。标量寄存器可以与 SPM 存储空间和 Global 存储空间交换数据。

**VPU**

VPU（Vector Processing Unit）向量处理单元，用于执行向量运算的部件，可以完成众多向量操作，包括向量存储操作、向量算术运算、向量类型转换、向量比较选择等。向量处理单元只能接受向量寄存器内的数据，在完成向量操作后将计算结果保存到向量寄存器内。

**VREG**

VREG（Vector Register）向量寄存器，用于存储向量数据的专用寄存器，支持多个标量数据元素的并行读写操作。向量寄存器可以与 SPM 存储空间和 Global 存储空间交换数据。

**FU**

FU（Function Unit）运算功能单元，可以完成矩阵乘等计算。运算功能单元只能接收 SPM 存储空间的数据，在完成运算后将计算结果保存到 SPM 存储空间。

**SPM 存储空间**

SPM 存储空间是计算核心 SPE 内私有的存储空间，有较高的访存带宽，但是可用容量较小，需要合理利用有限的 SPM 存储空间以提升运行程序的性能。SDAA C 将 SPM 存储空间进一步细分为：

- **堆空间**：可以与 FU、SREG、VREG、栈空间、local 空间以及 SPA 内 Global 存储空间交换数据。
- **栈空间**：可以与 FU、SREG、VREG、堆空间、local 空间以及 SPA 内 Global 存储空间交换数据。

  > **说明：**
  >
  > 在开启 `--stack-on-global` 选项，将函数栈从 SPM 存储空间切换到 Global 存储空间后，切换后的栈空间可以与 SREG、VREG、堆空间、local 空间以及 Global 存储空间交换数据。

- **local 空间**：可以与 FU、SREG、VREG、堆空间、栈空间以及 SPA 内 Global 存储空间交换数据。

**Global 存储空间**

Global 存储空间是计算核心阵列 SPA 内的内存空间，由计算核心阵列 SPA 内的所有计算核心 SPE 共享。Global 存储空间有较大的存储容量，但是访存带宽较小，可以通过数据搬运接口将数据搬运到 SPM 存储空间内进行计算，以提升程序性能。Global 存储空间可以与 SREG、VREG、堆空间、栈空间以及 local 空间交换数据。

### 4.6 数据传输

数据传输管理主要包括：主机端内存与设备端内存（Global 存储空间）之间的数据传递、同一计算核心阵列 SPA 的 Global 存储空间内的数据传递、Global 存储空间与 SPM 存储空间之间的数据传递、同一计算核心阵列 SPA 内不同计算核心 SPE 之间 SPM 存储空间的数据传递、以及同一计算核心 SPE 的 SPM 存储空间内的数据传递。

> 图1 数据传输

**主机端内存与 Global 存储空间之间的数据传递**

Kernel 函数在设备端运行时所需要的源数据，需要从主机端获取。在进行数据传输前，需要在设备端申请相应大小的 Global 存储空间，主机端将数据传输到已分配的 Global 存储空间，完成数据计算后，再将计算结果传回主机端内存，并释放申请的 Global 存储空间。

主机端内存与 Global 存储空间的数据传递：

| 操作 | 函数 |
|---|---|
| 主机端内存与 Global 存储空间的数据传递 | sdaaMemcpy，可参考 sdaaMemcpy。 |

**同一计算核心阵列 SPA 的 Global 存储空间内的数据传递**

同一计算核心阵列 SPA 内的 Global 存储空间和 Global 存储空间之间的数据传递，可参考 memcpy。

**Global 存储空间与 SPM 存储空间之间的数据传递**

为提升 SPE 的计算效率，需要将数据从 Global 存储空间传输到 SPM 存储空间进行计算。在进行数据传输前，需要在计算核心 SPE 上申请相应大小的 SPM 存储空间，设备端将数据传输到已分配的 SPM 存储空间进行计算，再将计算结果传回 Global 存储空间，并释放申请的 SPM 存储空间。

| 操作 | 函数 |
|---|---|
| Global 存储空间与 SPM 存储空间的数据传递 | memcpy，memcpy_async 等接口，可参考 DMA 数据搬运。 |

**不同计算核心 SPE 之间的数据传递**

为使不同计算核心 SPE 之间的数据交换更加方便，SDAA C 提供了 RMA 编程接口，用于搬运不同 SPM 存储空间之间的数据。RMA 编程接口实现了计算核心 SPE 之间的高效数据搬运，且可以缓解 SPM 存储空间容量受限的问题，可参考 RMA 数据搬运。

**在同一计算核心 SPE 的 SPM 存储空间内的数据传递**

计算核心 SPE 可在其内部的 SPM 空间进行数据传递，可参考 memcpy。

### 4.7 SDAARuntime 接口

仅在主机端编写 Kernel 函数，太初 AI 加速卡并不能执行计算任务：

1. 没有指定运行 Kernel 函数的目标计算核心阵列 SPA。
2. 没有申请设备端的 Global 存储空间。
3. 缺少计算所需的源数据。

因此，主机端通过 SDAARuntime 接口解决以上问题。SDAARuntime 接口可以：

1. 指定 Kernel 函数运行在设备端具体哪一个计算核心阵列 SPA 上，可参考 sdaaSetDevice。
2. 在设备端分配 Global 存储空间，申请 Global 存储空间可参考 sdaaMalloc；释放 Global 存储空间可参考 sdaaFree。
3. 将数据从主机端内存传递到设备端 Global 存储空间，可参考 sdaaMemcpy。

## 5. 语言规范

### 5.1 语法

#### 5.1.1 <<<...>>>

在主程序中使用 `<<<placeHolder, stream>>>` 语法调用 Kernel 函数，`<<<placeHolder, stream>>>` 共有两个参数：

```c
size_t placeHolder;   // 暂无实际含义
sdaaStream stream;    // SDAA流
```

其中：

- `placeHolder`：目前暂无实际含义，支持设置任意值。
- `stream`：SDAA 流（详细说明参考 SDAA 流（详情请参考《SDAARuntime 用户手册》的 SDAA 流章节）），可缺省。

> **说明：**
>
> 使用 `<<<...>>>` 调用 kernel 函数时，会同时调用 threadDim 个计算核心 SPE 进行计算，相当于同时启动 threadDim 个线程。

**使用示例**

```c
__global__ void user_kernel(int *a, int *b, int *c)
{
    // 根据实际情况，编写Kernel函数
    ...
}


int main()
{
    ...
    // 调用核函数进行计算
    sdaaStream_t stream;
    sdaaStreamCreate(&stream);
    user_kernel<<<1, stream>>>(a, b, c);
    // 第二个参数可以不写：user_kernel<<<1>>>(a, b, c);
    ...
}
```

### 5.2 关键字

#### 5.2.1 __host__

**功能介绍**

运行空间标识符，修饰 Host 端函数，在 Host 端调用、运行。

该关键字可以缺省。

**使用示例**

```c
__host__ void add(int *a, int *b, int *c)
{
    *c = *a + *b;
}


void calc(int *a, int *b, int *c)
{
    add(a, b, c);
}
```

#### 5.2.2 __global__

**功能介绍**

函数标识符，修饰 Kernel 函数。在 Host 端通过 `<<<...>>>` 调用，在 Device 端运行。

返回类型需要是 void。

**使用示例**

```c
__global__ void calc(int *a, int *b, int *c)
{
    ...
    *c = *a + *b;
    ...
}


int main()
{
    ...
    // 调用Kernel函数
    calc<<<1>>>(a, b, c);
    ...
}
```

#### 5.2.3 __device__

**功能介绍**

运行空间标识符，修饰 Device 端的函数或变量，在 Device 端调用、运行。可以被 `__device__` 或 `__global__` 修饰的函数调用。

**使用示例**

```c
__device__ void add()
{
    ...
}


__device__ void add2()
{
    ...
    add();
    ...
}


__global__ void calc()
{
    ...
    add2();
    ...
}
```

#### 5.2.4 __local__

**功能介绍**

变量标识符，修饰存放在 SPM 存储空间的变量。

只能作为全局变量使用，目前不支持初始化。可以通过 get_local_size 获取当前计算核心 SPE 可用 local 空间总大小。

**使用示例**

```c
__local__ int num[512];
__local__ short n;
```

#### 5.2.5 __scoped_local__

**功能介绍**

变量修饰符，修饰 Device 端函数栈变量。在开启 `--stack-on-global` 选项时，该关键字生效。生效后与 `__local__` 共享同一块 SPM 存储空间，可以通过 get_local_size 获取当前计算核心 SPE 可用 local 空间总大小。

> **说明：**
>
> 当未启用 `--stack-on-global` 选项时，所有栈变量默认位于 SPM 存储空间中。
>
> 在这种情况下，编译器将忽略 `__scoped_local__` 关键字，因此被 `__scoped_local__` 修饰的变量将被处理为普通栈变量。

被 `__scoped_local__` 修饰的变量类似于普通的栈变量，在每个线程内，该变量：

- 只能在函数内访问。
- 具有函数内的生命周期。
- 支持初始化，且每次函数调用时初始化，若没有显式初始化，其初始值是未定义的。

但也与栈变量有重要区别，该变量：

- 驻留在 SPM 存储空间中，而不是在栈上。
- 拥有静态存储位置，其地址在程序开始运行时确定，并在程序结束前一直保持不变。
- 不支持在递归函数内使用此变量。

**使用示例**

```c
// __global__ 或者 __device__ 修饰的Device端函数
__device__ void func()
{
    __scoped_local__ int num[512];
    __scoped_local__ short n = 1;
    ...
}
```

#### 5.2.6 threadIdx

**功能介绍**

`threadIdx` 用于获取 Device 端计算核心 SPE 的 ID，可以为每一个计算核心 SPE 定制不同的行为。

**使用示例**

```c
__device__ void user_kernel()
{
    if (threadIdx == 0) {
          ...
    } else if (threadIdx == 1) {
          ...
    } else {
         ...
    }
}
```

#### 5.2.7 threadDim

**功能介绍**

`threadDim` 用于获取同一计算核心阵列 SPA 内计算核心 SPE 的总数。

**使用示例**

```c
__device__ void user_kernel()
{
    unsigned long thread_count = threadDim;
}
```

### 5.6 常用 SDAARuntime 接口

SDAARuntime 接口可以对太初 AI 加速卡进行设备管理、资源申请和资源释放等。

在 SDAA C 中常用的 SDAARuntime 接口如下表，更多且详细的 SDAARuntime 接口描述可参考 SDAARuntime 用户手册（详情请参考《SDAARuntime 用户手册》的 API 参考章节）：

| SDAA 接口 | 功能介绍 |
|---|---|
| sdaaDeviceReset | 销毁当前进程中当前设备上的所有已分配的资源并重置所有状态。 |
| sdaaSetDevice | 指定 Kernel 函数运行在 Device 端具体哪一个计算核心阵列 SPA 上。 |
| sdaaGetDevice | 获取当前线程使用的设备。 |
| sdaaGetDeviceCount | 获取设备的数量。 |
| sdaaDeviceSynchronize | 等待当前线程使用的设备完成之前的所有请求任务。 |
| sdaaMalloc | 申请设备端 Global 存储空间。 |
| sdaaFree | 释放设备端 Global 存储空间。 |
| sdaaMemcpy | 在主机端内存和设备端内存（Global 存储空间）之间传输数据。 |
| sdaaMemset | 将 Global 存储空间的起始地址初始化为一个特定值。 |
| sdaaStreamCreate | 创建一个新的异步流。 |

**使用示例**

```c
__global__ void calc(long *a)
{
    // 根据实际情况，编写Kernel函数
    ...
}


int main()
{
    int dev_count = 0;
    long a = 0;
    long *dev_a = NULL;
    // 1. 判断是否有可用设备
    sdaaGetDeviceCount(&dev_count);
    assert(0 < dev_count);
    int device_id = 0;
    // 2. 设置使用的设备
    sdaaSetDevice(device_id);
    // 3. 申请设备端Global存储空间
    sdaaMalloc((void**)(&dev_a), sizeof(long));
    // 4. 初始化Global存储空间
    sdaaMemset(dev_a, 0, sizeof(long));
    // 5. 从Host端内存拷贝数据到Device端的Global存储空间
    sdaaMemcpy(dev_a, &a, sizeof(long), sdaaMemcpyHostToDevice);
    // 6. 调用Kernel函数
    calc<<<1>>>(dev_a);
    // 7. 等待Kernel函数执行结束
    sdaaDeviceSynchronize();
    // 8. 从Device端的Global存储空间拷贝计算结果到Host端内存
    sdaaMemcpy(&a, dev_a, sizeof(long), sdaaMemcpyDeviceToHost);
    // 9. 释放设备端Global存储空间
    sdaaFree(dev_a);
    return 0;
}
```

## 7. 函数接口

### 7.1 概述

函数接口是编译器内部实现的函数，SDAA C 内置了计算核心 SPE 同步、SPM 内存分配和释放、数据搬运、原子操作、矩阵乘、向量操作和性能采样操作等函数接口，为用户提供简单、易用、功能丰富的编程接口，从而更高效地利用太初 AI 加速卡的强大算力。

### 7.2 线程组

#### 7.2.1 概述

线程组是指对计算核心阵列 SPA 中的所有计算核心 SPE 进行组合，根据需要将若干个计算核心 SPE 组合为一个线程组，您可以对线程组内的计算核心 SPE 执行计算核心 SPE 同步、Broadcast 数据搬运等操作。线程组由以下函数接口组成：

- **ThreadGroup**：用于配置线程组，可以将指定的计算核心 SPE 组合为一个线程组。
- **thread_group_set_mask**：以掩码的方式设置当前 SPA 内的各个 SPE 是否加入目标线程组。
- **thread_group_include**：在目标线程组内添加指定的计算核心 SPE。
- **thread_group_exclude**：在目标线程组内删除指定的计算核心 SPE。
- **thread_group_is_included**：判断指定的计算核心 SPE 是否在指定的线程组内。
- **thread_group_get_size**：获取指定线程组内计算核心 SPE 的数量。
- **thread_group_clear**：删除目标线程组内的所有计算核心 SPE。

```c

ThreadGroup(unsigned long *thread_group, unsigned int size)
ThreadGroup(unsigned long mask, unsigned long position = 0)
```

**功能介绍**

**参数解释**

| 参数名称 | 说明 |
|---|---|
| thread_group | 线程组，由计算核心 SPE 的 ID 组成。 |
| size | 参数 thread_group 数组的长度。<br>**说明**：size 的最大长度为计算核心阵列 SPA 内计算核心 SPE 的数量，即 threadDim。 |

### 7.3 计算核心 SPE 同步

#### 7.3.1 概述

计算核心 SPE 同步可以对同一计算核心阵列 SPA 内的所有或者部分计算核心 SPE 的执行进度进行同步。计算核心 SPE 同步由以下函数接口组成：

- **sync_threads**：对同一计算核心阵列 SPA 内的所有或者部分计算核心 SPE 进行同步操作。

#### 7.3.2 sync_threads

```c
void sync_threads()
void sync_threads(const ThreadGroup &thread_group)
```

**功能介绍**

对同一计算核心阵列 SPA 内的所有或部分计算核心 SPE 进行同步操作，即当所有或设置的计算核心 SPE 都运行到 `sync_threads` 时才会执行下一条语句：

- 不携带参数：对同一计算核心阵列 SPA 内的所有计算核心 SPE 进行同步操作。
- 携带参数：对同一计算核心阵列 SPA 内由线程组指定的计算核心 SPE 进行同步操作。

**参数解释**

| 参数名称 | 说明 |
|---|---|
| thread_group | ThreadGroup 类型，标识指定的线程组，包含需要同步的所有 SPE 的 threadIdx。 |

**返回值**

无。

**注意事项**

- 需要引入 sdaa 命名空间。
- 不携带参数：`sync_threads` 函数需要在当前所有 SPE 都可以执行到的代码段内执行。
- 携带参数：`sync_threads` 函数需要在指定的 SPE 都可以执行到的代码段内执行。

**使用示例 1**

```c
#define SIZE 100


__local__ int input[SIZE];
__global__ void func()
{
    // 为所有的计算核心SPE初始化input数组
    for (int i = 0; i < SIZE; i++) {
        input[i] = i * threadIdx;
    }


    // 在当前所有计算核心SPE都可以执行到的代码段内执行同步操作，以确保当所有计算核心SPE执行到该位置时，才会执行下一条语句
    sdaa::sync_threads();
}
```

**使用示例 2**

```c
using namespace sdaa;
#define SIZE 100
#define THREAD_GROUP_SIZE 6


__local__ int input[SIZE];
__global__ void func()
{
    // 为所有的计算核心SPE初始化input数组
    for (int i = 0; i < SIZE; i++) {
        input[i] = i * threadIdx;
    }


    // 表示0，5，9，12，18，22六个线程进行同步操作
    unsigned long arr[] = {0, 5, 9, 12, 18, 22};
    // 创建线程组
    ThreadGroup thread_group(arr, THREAD_GROUP_SIZE);
    // 对指定的部分计算核心SPE执行同步操作
    sync_threads(thread_group);
}
```

**使用示例 3**

```c

#define SIZE 100
#define THREAD_GROUP_SIZE 6


__local__ int input[SIZE];


__global__ void func()
{
    // 为所有的计算核心SPE初始化input数组
    for (int i = 0; i < SIZE; i++) {
        input[i] = i * threadIdx;
    }


    // 表示0，5，9，12，18，22六个线程进行同步操作
    unsigned long arr[] = {0, 5, 9, 12, 18, 22};
    // 创建线程组
    ThreadGroup thread_group(arr, THREAD_GROUP_SIZE);


    // 对指定的部分计算核心SPE执行同步操作，仅需要同步的计算核心SPE执行同步操作。
    if (thread_group_is_included(thread_group, threadIdx)) {
        sync_threads(thread_group);
    }
}
```

### 7.4 SPM 内存分配和释放

#### 7.4.1 概述

SPM 内存分配和释放接口可以对 SPM 内存进行精准管理。SPM 内存分配和释放接口主要由以下函数接口组成：

- **malloc**：根据对 SPM 存储空间的使用需求，判断 SPM 存储空间是否可用，并根据判断结果返回失败或可用空间的起始地址。
- **free**：对于给定的 SPM 存储空间地址，释放以此地址为起始位置的 SPM 存储空间。

#### 7.4.2 malloc

```c
void *malloc(size_t size, MallocDirect direct = AddressLowToHigh)
```

**功能介绍**

根据给定的内存大小和分配方向，判断 SPM 存储空间中是否有空间满足需求。如果满足，则分配 SPM 存储空间并返回该段空间的起始地址；否则返回零地址。

**参数解释**

| 参数名称 | 说明 |
|---|---|
| size | 需要分配的 SPM 存储空间大小，单位：Byte。 |
| direct | 枚举类型 MallocDirect，支持以下两种内存分配方向：<br>AddressLowToHigh：自低地址向高地址方向分配，默认内存分配方向。<br>AddressHighToLow：自高地址向低地址方向分配。 |

**返回值**

| 返回值类型 | 说明 |
|---|---|
| void * | 如果 SPM 中有空间可以满足本次分配需求，返回可用空间的起始地址。否则，返回零地址。 |

**注意事项**

- 用户的 SPM 需求空间会自动 64B 对齐。例如，用户需求 4B 的 SPM 存储空间，接口会按 64B 判断并分配；用户需求 100B 的 SPM 存储空间，接口会按 128B 判断并分配。
- 该接口可分配的最大 SPM 存储空间为 0x3AC00 字节。
- 对于从低地址向高地址分配的需求，接口分配的起始地址为 0x1400。对于从高地址向低地址分配的需求，接口分配的起始地址为 0x3C000。
- `malloc` 一般与 free 配合使用。

**使用示例**

调用 `malloc` 分别从 SPM 低地址和高地址开始分配 4B 大小的 SPM 存储空间，使用后调用 free 释放这部分存储空间。

```c
__device__ void test()
{
      size_t size = 4;
      // 申请4个Byte的空间，分配给指针addr_low
      void *addr_low = malloc(size);
      // 地址从高到底申请4个Byte的空间，分配给指针addr_high
      void *addr_high = malloc(size, AddressHighToLow);


      // 使用addr_low, addr_high存储空间
      ...


      // 释放申请的SPM空间的指针
      free(addr_low);
      free(addr_high);
}
```
