## 17. 常见问题

### 17.1 如何使用 CMake 构建？

为使 CMake 可以构建后缀为 `.scpp` 的文件，需要先使用 `set_source_files_properties` 命令，设置 `LINKER_LANGUAGE` 的属性为 `CXX`。CMake 构建分为 LTO 模式和非 LTO 模式。

- **LTO 模式**：在生成动态库或者可执行文件时，用到的所有 `.scpp` 文件都在同一个编译模块里，即：一个 `.scpp` 里定义的变量和函数，可以在另一个 `.scpp` 文件里使用。
- **非 LTO 模式**：不同的 `.scpp` 文件是不同的编译模块。在不同编译模块中的设备端代码，相互之间没有依赖关系，即：在一个编译模块里定义的变量和函数，在另一个编译模块里不能对其进行访问。

#### 17.1.1 LTO 模式

##### 17.1.1.1 示例代码

- `main2.scpp` 使用 `main1.scpp` 定义的 `__device__ void print()`。
- `main.cpp` 链接生成的静态库或者动态库，生成可执行文件。

```cpp
//main1.scpp
#include <stdio.h>
#include <stdlib.h>


__device__ void print()
{
    printf("this is device function in %s\n", __FILE__);
}


__global__ void test1(int a, double b)
{
    print();
    printf("this is %s!\n", __FILE__);
    printf("a = %d, b = %lf\n", a, b);
    printf("\n");
}


int main1()
{
    sdaaSetDevice(0);


    test1<<<1>>>(1, 3.0);


    sdaaDeviceSynchronize();


    return 0;
}
```

```cpp
//main2.scpp
#include <stdio.h>
#include <stdlib.h>


extern __device__ void print();


typedef struct {
    int b;
    int c;
} st1;
st1 a = {1, 2};


__global__ void test2(int a, double b, st1 c)
{
    print();
    printf("this is %s!\n", __FILE__);
    printf("a = %d, b = %lf, c.b = %d, c.c = %d\n", a, b, c.b, c.c);
    printf("\n");
}


int main2()
{
    sdaaSetDevice(0);


    test2<<<1>>>(66, 8.0, a);


    sdaaDeviceSynchronize();


    return 0;
}
```

```cpp
//main.cpp
extern int main1();
extern int main2();



int main()
{
    main1();
    main2();
}
```

##### 17.1.1.2 静态库

1. 编写 `CMakeLists.txt` 文件。

   > 说明：
   >
   > 1. 指定 `add_library` 为 `OBJECT` 类型。
   > 2. 调用 `add_custom_target` 自定义 target 生成静态库，需要使用 `-flto --sdaa-link -shared -sdaa-static-lib -fuse-ld=lld` 链接选项。

   ```cmake
   cmake_minimum_required(VERSION 3.13)


   project (lto-static-lib)


   #设置编译器为TecoCC
   set(CMAKE_CXX_COMPILER "tecocc")


   #libtmp
   #步骤1：生成objs
   file(GLOB_RECURSE SRC_TMP "*.scpp")
   add_library(tmp_objs OBJECT ${SRC_TMP})
   set_source_files_properties(${SRC_TMP} PROPERTIES LANGUAGE CXX)
   target_compile_options(tmp_objs PRIVATE -flto)


   #步骤2：生成lib
   #ALL, 默认构建libtmp，否则可能只生成objs
   add_custom_target(tmp ALL
         COMMAND tecocc $<TARGET_OBJECTS:tmp_objs> -flto --sdaa-link -shared -sdaa-static-lib -fuse-ld=lld
         -o libtmp.a
         COMMAND_EXPAND_LISTS
   )
   add_dependencies(tmp tmp_objs)


   #生成可执行文件，链接libtmp
   #tmp是custom target，通过add_dependencies添加依赖
   add_executable(static_test main.cpp)
   add_dependencies(static_test tmp)
   target_link_libraries(static_test PRIVATE -L. -ltmp -fuse-ld=lld)
   ```

2. 执行以下命令，创建并进入 build 目录。

   ```bash
   mkdir build && cd build
   ```

3. 执行以下命令，生成 Makefile 文件。

   ```bash
   cmake ..
   ```

4. 执行以下命令，生成可执行文件。

   ```bash
   make -j2
   ```

### 17.2 链接 LTO 模式编译的库报错，`libxxx.a: archive has no index; run ranlib to add one`，如何解决？

**根因分析**

通常是使用 AR 工具将目标文件打包成 `.a` 静态库文件 `libxxx.a` 所导致。

**解决方案**

- 如果静态库中只包含 Device 端代码：
  - 编译 `.o` 文件时，增加 `--sdaa-device-only` 选项。
  - 链接 `libxxx.a` 时，增加 `--sdaa-static-device-lib=libxxx.a` 选项。
- 如果同时包含 Device 端和 Host 端的代码：不使用 AR 工具将目标文件打包成 `.a` 静态库文件 `libxxx.a`，推荐用法参考 LTO 模式静态库。

### 17.3 CUDA 程序迁移

#### 17.3.1 如何解决线程编号不对应？

**问题描述**

CUDA 编程时，线程编号是通过 `gridDim`、`blockIdx`、`blockDim`、`threadIdx` 确定的，但在 SDAA C 编程中只有 `threadDim` 和 `threadIdx`，两者该如何进行对应？

例如：以下代码中每次 `i` 相加的步长该如何进行对应？

```c
for (size_t i = blockIdx.x * blockDim.x + threadIdx.x; i < (size); i += blockDim.x * gridDim.x)
```

**解决方案**

CUDA 编程模型中线程层级为 grid-block-thread，SDAA C 在单个计算核心阵列 SPA 中只使用 thread 粒度，对应计算核心 SPE，简化并行层级。

在单 SPA 编程中，对线程的标识为预定义变量 `threadIdx`。通过将任务划分为子任务，每个线程在运行相同程序时执行不同子任务，此即 SPMD（Single Program Multiple Data，单程序多数据）编程范式。

以上 CUDA 程序改写为 SDAA C 程序如下：

```c
for (size_t i = threadIdx; i < (size); i += threadDim)
```

#### 17.3.2 如何解决调用 Kernel 函数不对应？

**问题描述**

CUDA 调用 Kernel 函数需要指定 block id 和 thread id，但使用 SDAA C 调用 Kernel 函数只有 `<<<placeHolder, stream>>>`，两者如何对应？

例如：以下代码如何修改为 SDAA C 的程序？

```c
MomentumUpdateVariableKernel<<<GET_BLOCKS(size), GET_THREADS, 0, cuda_stream>>>
```

**解决方案**

SDAA C 中 Kernel 函数调用单计算核心阵列 SPA，分配固定的硬件资源，即会同时调用同一个 SPA 内的 `threadDim` 个计算核心 SPE 进行计算，相当于同时启动 `threadDim` 个线程。

以上 CUDA 程序修改为 SDAA C 程序时，保留 `placeHolder` 和 `stream` 参数即可：

```c
MomentumUpdateVariableKernel<<<0, sdaa_stream>>>
```

#### 17.3.3 SDAA C 中是否有与 CUDA 对应的线程束 Warp？

**问题描述**

使用 CUDA 进行编程时，有线程束 Warp 的概念，SDAA C 中是否有与之对应的概念？

例如：在以下 CUDA 程序中，改写为 SDAA C 程序时，`warp_start_row` 该如何处理？

```c
int shm_row_id = (threadIdx >> 5);
int shm_col_id = (threadIdx % 32);
int block_start_row = blockIdx.x * rows_per_block;
int block_end_row = block_start_row + rows_per_block;
block_end_row = block_end_row < m ? block_end_row : m;
int warp_start_row = blockIdx.x * rows_per_block + shm_row_id * rows_per_warp;
int warp_end_row = warp_start_row + rows_per_warp;
int real_rows_per_warp = warp_end_row < block_end_row ? rows_per_warp : block_end_row - warp_start_row;
```

**解决方案**

SIMT 的架构是以 Warp 为单位，可以由一个指令一次驱动 32 个 thread，并且多 Blocks、多 Warps 可以并行计算。

由于 SDAA C 使用 SPMD 架构，与 SIMT 架构并不相同。在 SDAA C 中，没有 Block 和 Warp 的相关概念。

在 SDAA C 中，一个计算核心阵列 SPA 内有 `threadDim` 个计算核心 SPE（类比 CUDA 的 Thread）是并行计算的。可以使用 `threadIdx`，获取 Device 端计算核心 SPE 的 ID，为每一个计算核心 SPE 定制不同的行为。

#### 17.3.4 如何在 Kernel 函数中调用数学函数接口？

数学函数不需要 `#include` 相应的头文件即可使用。

数学函数的接口信息，可参考数学函数的详细内容；数学库的使用示例，可参考接口使用示例。

#### 17.3.5 CUDA 的 blockId 如何修改为 SDAA C 代码？

SDAA C 没有 Grid, Block 等概念。

在 SDAA C 中，启动 Kernel 函数时，默认会启动 `threadDim` 个计算核心，您可以通过 `threadIdx` 获取当前正在执行任务的计算核心 SPE 的 ID，并为每个计算核心 SPE 定制不同的行为。

#### 17.3.6 CUDA 的 share memory 如何修改为 SDAA C 代码？

SDAA C 中没有提供 share memory，可以用的存储空间包括 Global 存储空间和 SPM 存储空间：

- Global 存储空间，是计算核心阵列 SPA 的共享存储。
- SPM 存储空间，是计算核心 SPE 的私有空间。

对于计算核心 SPE 而言，直接访问 Global 存储空间，速度缓慢。在 SPE 执行计算任务的过程中，如果从 SPM 存储空间获取数据，计算效率将得到显著提升。

您可以通过 RMA 数据搬运实现同一计算核心阵列 SPA 内两个不同计算核心 SPE 的 SPM 存储之间的数据流动。

### 17.4 使用 VSCode 打开 SDAA C 程序时，如何实现代码高亮、代码自动补全和代码高亮提醒等功能？

**问题描述**

使用 VSCode 打开 SDAA C 程序时，无法使用代码高亮、代码自动补全和代码高亮提醒等功能。

**解决方案**

1. 单击页面左下角的 > 设置。

2. 在弹出的页面中，选择目标远程端口页签。

3. 单击页签右上角的图标。

4. 在 `setting.json` 文件中，插入以下字段。

   ```json
   "files.associations": {
       "scpp": "cpp"
    },
    "C_Cpp.default.forcedInclude": [
       "/opt/tecoai/extras/llvm/lib/clang/11.0.0/include/__clang_sdaa_runtime_wrapper.h",
    ],
    "C_Cpp.default.includePath": [
       "/opt/tecoai/include/",
       "/opt/tecoai/extras/llvm/lib/clang/11.0.0/include/",
    ],
    "C_Cpp.default.defines": [
       "__SDAA__",
       "__clang__"
    ],
   ```

5. 保存退出。

**执行结果**

使用 VSCode 打开 SDAA C 程序时，可以正常使用代码高亮、代码自动补全和代码高亮提醒等功能。
