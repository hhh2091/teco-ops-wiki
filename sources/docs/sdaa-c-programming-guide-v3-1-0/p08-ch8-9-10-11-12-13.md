#### 9.2.2 batch_sin

```c
template<typename T>
void batch_sin(T *dst, const T *src, int len)
```

**功能介绍**

批量计算输入数组 src 的正弦值，并将计算结果存储到 dst 数组中。

**参数解释**

| 参数名称 | 说明 |
|---------|------|
| dst | 输出数组。<br>说明：<br>当输入数组 src 中某个数据为 0 时，对应位置返回 0。<br>当输入数组 src 中某个数据为 ±∞ 时，对应位置返回 NaN。<br>当输入数组 src 中某个数据为 NaN 时，对应位置返回 NaN。 |
| src | 输入数组。 |
| len | 数组的长度。 |

**返回值**

无。

**注意事项**

- 需要手动包含 scl.h 头文件。
- 该接口仅支持 float 数据类型和 half 数据类型。
- 需要引入 sdaa::scl 命名空间。

**使用示例1**

```c
#include <scl.h>


__device__ void test(float *dst, float *src, int len){
     sdaa::scl::batch_sin(dst, src, len);
}
```

**使用示例2**

```c
#include <scl.h>


__device__ void test(half *dst, half *src, int len){
     sdaa::scl::batch_sin(dst, src, len);
}
```

#### 9.2.5 batch_pow

```c

void batch_pow(T *dst, const T *src1, const T *src2, int len)
```

**功能介绍**

**参数解释**

| 参数名称 | 说明 |
|---------|------|
| dst | 输出数组。<br>说明：<br>当输入数组 src1 中某个数为 +∞ 时：<br>　如果 src2 中对应位置的数小于 0，对应位置返回 0。<br>　如果 src2 中对应位置的数大于 0，对应位置返回 +∞。<br>当输入数组 src1 中某个数为 -∞ 时：<br>　如果 src2 中对应位置的数小于 0，对应位置返回 0。<br>　如果 src2 中对应位置的数大于 0 且小数位不为 0 时，对应位置返回 +∞。<br>　如果 src2 中对应位置的数大于 0 且小数位为 0 时，对应位置返回 -∞。<br>当输入数组 src2 中某个数为 +∞ 时：<br>　如果 src1 中对应位置的数属于 (-1, 1)，对应位置返回 0。<br>　如果 src1 中对应位置的数的绝对值为 1，对应位置返回 NaN。<br>　如果 src1 中对应位置的数不属于以上两种情况，对应位置返回 +∞。<br>当输入数组 src2 中某个数为 -∞ 时：<br>　如果 src1 中对应位置的数属于 (-1, 1)，对应位置返回 +∞。<br>　如果 src1 中对应位置的数的绝对值为 1，对应位置返回 NaN。<br>　如果 src1 中对应位置的数不属于以上两种情况，对应位置返回 0。<br>如果 src1 或者 src2 中某个数为 NaN 时，对应位置返回 NaN。 |
| src1 | 输入数组。 |
| src2 | 输入数组。 |
| len | 数组的长度。 |

**返回值**

无。

**注意事项**

- 需要手动包含 scl.h 头文件。

#### 9.2.8 batch_axpy

```c
template<typename T>
```

| dst | 输出数组。<br>说明：<br>输入数组 src1 或者 src2 中某个元素为 NaN，或者输入系数 alpha 为 NaN 时，对应位置输出 NaN。<br>输入数组 src1 或者 src2 中某个元素为 ±∞，或者输入系数 alpha 为 ±∞ 时，对应位置输出 ±∞。 |
| src1 | 输入数组。 |
| src2 | 输入数组。 |
| alpha | 输入系数。 |
| len | 数组的长度。 |

**返回值**

无。

**注意事项**

- 需要手动包含 scl.h 头文件。
- 该接口仅支持 float 数据类型和 half 数据类型。
- 需要引入 sdaa::scl 命名空间。

**使用示例1**

```c
#include <scl.h>


__device__ void test(float *dst, float *src1,   float *src2, T alpha, int len)
```

### 10.1 命令行选项

TecoCC 是一个能自动为编译的目标文件传递适当的头文件及库文件的编译器驱动程序。TecoCC 提供了简单且常用的命令行选项，可以简化 C++ 及 SDAA C 代码的编译过程并调用不同编译阶段的工具集合执行编译流程。

| 选项 | 功能 |
|------|------|
| -h / --help | 显示 SDAA C 支持的所有命令行选项的概要帮助信息，包括命令行选项的基本用法和参数列表。 |
| -flto | 开启 LTO 优化模式。 |
| -v | 显示编译器的版本。 |
| --verbose | 显示编译器内部编译各过程的命令行的详细信息。 |
| --sdaa-compile-host-device | 编译 SDAA C 的 Host 端及 Device 端代码（默认）。 |
| --sdaa-device-lib-path=\<value\> | 指定 SDAA C 的 Device 端库的搜索路径。 |
| --sdaa-device-lib=\<value\> | 指定链接 SDAA C 的 Device 端 .bc 库。 |
| --sdaa-static-device-lib=\<value\> | 指定链接 SDAA C 的 Device 端静态库。 |
| --sdaa-host-only | 编译 SDAA C 代码时，指定只编译 Host 端代码。 |
| --sdaa-device-only | 编译 SDAA C 代码时，指定只编译 Device 端代码。 |
| --sdaa-link | 链接 SDAA C 的 clang-offload-bundler 捆绑包。 |
| -sdaa-new-launch-api | 指定开启 SDAADriver v0.6.0 及之后至 SDAADriver v0.9.0 之前版本 Kernel 函数的 Launch 接口。<br>说明：适用于 TecoCC v1.3.0 及之后至 v1.3.4 之前的版本。 |

| -nosdaalib | SDAA C 的 Device 端链接时，不链接默认链接的 Device 端库文件。 |
| --stack-on-global | SDAA C 将函数栈从 SPM 存储空间切换到 Global 存储空间。<br>说明：<br>若对代码分阶段编译，则所有阶段必须都开启该选项。<br>同一可执行文件下的所有函数，必须同步选择是否开启，部分开启而部分不开启会导致未定义错误。 |
| --sdaa-arch=pcx_100 | 用于生成《PCX 编程指南》相关代码。 |

### 10.2 编译流程

#### 10.2.1 LTO 模式

LTO 模式（Link Time Optimization）即链接时优化模式，也称模块间优化，是指在链接时进行的跨模块的优化，推迟代码生成到链接时，可以进行内联或全局优化。

在打开 -flto 选项（可参考支持的命令行选项）时，即开启 LTO 优化模式，编译流程如下（详细的编译代码，可参考编译示例1）：

1. 剥离源文件中的 Host 端代码，经过预处理、编译阶段生成 Host 端的 .bc 文件。
2. 剥离 Device 端的代码，经过预处理、编译阶段生成 Device 端的 .bc 文件。
3. （可选）如果通过 -c 进行编译，编译器会将源文件中 Host 端和 Device 端的 .bc 文件打包成一个 .o 文件（实质是 .bc 文件）。此时若做链接，会将编译生成的 .o 文件拆解成 .bc 文件，再进行步骤 4 的处理。
4. 链接器将多个 Device 端的 .bc 文件链接成一个 Device 端的二进制文件，将这个二进制文件处理成 Host 端的 .bc 文件，再与其他 Host 端的 .bc 文件一起做链接，最终生成 Host 端的二进制文件。

图1 LTO 模式的编译流程

#### 10.2.2 非 LTO 模式

非 LTO 模式是普通的编译模式，编译生成的 .o 文件是 Linux 下 GNU 兼容的 .o 文件，可以使用 GCC 链接器进行链接，编译流程如下（详细的编译代码，可参考编译示例1）：

1. 剥离将源文件中的 Device 端程序，经过预处理、编译、链接阶段，最终生成 Device 端的二进制文件。
2. 剥离 Host 端的代码，在 Host 端的代码进行前端编译时嵌入 Device 端的二进制，经过 Host 端的编译、汇编、链接阶段最终生成 Host 端的二进制文件。

> 说明：
> Host 端和 Device 端的编译流程均分为预处理、编译、汇编、链接四个阶段。

### 10.3 编译方式

#### 10.3.1 概述

TecoCC 有三种编译方式：动态链接、静态链接和直接生成可执行文件。

- 动态链接和静态链接：主要应用于 main 函数单独存放于 .cpp 文件中的场景，可参考编译示例2。
- 直接生成可执行文件的使用场景没有限制，main 函数存放于某个 .scpp 文件或单独存放于 .cpp 文件的场景均适用。

#### 10.3.2 LTO 模式

##### 10.3.2.1 动态链接方式

1. 执行以下命令，将 .scpp 文件编译成 .o 文件，详细的编译代码，可参考编译示例2。

```bash
# 将源文件编译成.o文件
# 使用-flto选项开启LTO优化模式
# 这里的.o文件实际上是.bc文件
# -On指定优化度的级别，本例使用O2级别，您可以根据实际需求使用-O0/1/2/3，数值越大，级别越高。
tecocc user_kernel1.scpp -O2 -flto -c -o user_kernel1.o
tecocc user_kernel2.scpp -O2 -flto -c -o user_kernel2.o
```

2. 执行以下命令，将多个 .o 文件链接成动态库。

```bash
# 使用tecocc将所有的.o文件链接成动态库
# --sdaa-link选项指定链接SDAA C的clang-offload-bundler捆绑包

```


## 11 程序调试

### 11.1 概述

程序调试是应用程序开发人员在进行软件开发过程中必要的纠错手段。一款优秀的程序调试工具可以协助软件开发人员快速定位代码在编译、运行时发生错误的原因和位置，提升软件的开发效率。

SDAA C 为应用程序开发人员提供了配套的基本软件调试手段：

- 通过打印运行时日志的方式，追踪打印计算核心 SPE 中代码的运行状态，从而定位出问题发生的具体原因，可参考打印运行时日志。
- 通过 TecoGDB 工具调试计算核心 SPE 的代码，从而定位出问题发生的具体原因，可参考 TecoGDB 调试工具。
- 通过终止设备端程序的方式，在设备端的库函数代码中检测到无法修复的问题时，及时停止设备端程序的运行，可参考终止设备端程序。
- 通过断言的方式，检测代码中存在的错误，出现错误时打印对应的文件行号等详细信息，可参考断言。

### 11.2 打印运行时日志

打印运行时日志是 SDAA C 提供的一个代码追踪手段。用户可以通过调用 printf 接口打印目标变量的运行时状态。

**使用方法**

- printf 接口是 SDAA C 提供的内置接口，用户使用时无需引入头文件。
- printf 接口参数与 C/C++ 提供的 printf 接口相同。
- 设置环境变量 `export SDAA_SYNC_PRINT=1`，可以按照 SPE 编号从小到大的顺序进行打印。

**使用示例**

以打印参数的相加结果为例，示例代码如下：

```c
__device__ void sum_func(int a, int b)
{
    int g_sum = a + b;
    printf("threadIdx: %lu, sum: %d\n", threadIdx, g_sum);
}


__global__ void bar()
{
    sum_func(1, 2);
}


int main()
{
    sdaaSetDevice(0);
    bar<<<1>>>();


    sdaaDeviceSynchronize();


    return 0;
}
```

- 不设置环境变量 SDAA_SYNC_PRINT 或者值默认为 0 时，会按照 SPE 的执行顺序进行打印。执行后打印结果如下：

```
threadIdx: 4, sum: 3
threadIdx: 0, sum: 3
threadIdx: 6, sum: 3
threadIdx: 7, sum: 3
......
```

- 设置环境变量 SDAA_SYNC_PRINT=1，按照 SPE 编号从小到大的顺序进行打印。执行后打印结果如下：

```
threadIdx: 0, sum: 3
threadIdx: 1, sum: 3
threadIdx: 2, sum: 3
threadIdx: 3, sum: 3
......
```

### 11.3 TecoGDB 调试工具

TecoGDB（Tecorigin GNU Debugger）是基于 GNU GDB 开发的运行在 Linux 系统上的源码级调试工具，可以调试运行在太初 AI 加速卡上的程序，提供对被调试程序的运行控制和信息打印功能。该工具提供真实物理环境下的调试功能，避免仿真调试和真实环境运行的误差，方便您准确定位问题。

更多内容，可参考《TecoGDB 命令行工具用户手册》。

### 11.4 终止设备端程序

终止设备端程序是 SDAA C 提供的协助调试手段。如果您在设备端编写的库函数代码中检查到无法修复的问题，可以通过 abort 接口直接终止设备端的程序运行。通过 abort 函数结束运行时：

- 如果设置了环境变量 SDAA_ENABLE_COREDUMP_ON_EXCEPTION（详情请参考《SDAARuntime 用户手册》的 SDAA_ENABLE_COREDUMP_ON_EXCEPTION 章节）：通知主机端生成包含错误相关上下文的 Core Dump 文件，并且终止主机端的代码执行。
- 如果没有设置该环境变量 SDAA_ENABLE_COREDUMP_ON_EXCEPTION（详情请参考《SDAARuntime 用户手册》的 SDAA_ENABLE_COREDUMP_ON_EXCEPTION 章节）：继续执行主机端的代码，后续通过执行 sdaaDeviceSynchronize 接口，获取对应的 abort 错误码。

**使用方法**

- abort 接口是 SDAA C 提供的内置接口，使用时无需引入头文件。
- abort 接口的参数与 C/C++ 提供的 abort 接口和用法相同。

**使用示例**

以检测除数不为 0 作为示例，当检查到除数为 0 时，执行 abort 终止 SPE 后续程序的运行。

```c
__device__ int g_sum = 0;
__device__ int my_div(int a, int b)
{
    if (b == 0) {
        // 如果除数为0，终止SPE后续程序的运行
        abort();
    }
    return a / b;
}


__device__ void my_compute(int a, int b)
{
    int div_result = my_div(a, b);
    g_sum = (a - b) * div_result;
}


__global__ void bar()
{
    my_compute(1, 2);
}


int main()
{
    sdaaSetDevice(0);
    bar<<<1>>>();
    sdaaDeviceSynchronize();


    return 0;
}
```

### 11.5 断言

断言（assert）是 SDAA C 提供的检测错误手段。您可以通过调用 assert 接口检查代码中是否出现错误，出现错误后会和 C/C++ 提供的 assert 一样，打印对应错误所在的文件行号等信息，之后终止设备端程序的运行。通过 assert 函数终止设备端程序运行时：

- 如果设置了环境变量 SDAA_ENABLE_COREDUMP_ON_EXCEPTION（详情请参考《SDAARuntime 用户手册》的 SDAA_ENABLE_COREDUMP_ON_EXCEPTION 章节）：通知主机端生成包含错误相关上下文的 Core Dump 文件，并且终止主机端的代码执行。
- 如果没有设置环境变量 SDAA_ENABLE_COREDUMP_ON_EXCEPTION（详情请参考《SDAARuntime 用户手册》的 SDAA_ENABLE_COREDUMP_ON_EXCEPTION 章节）：继续执行主机端的代码，后续通过执行 sdaaDeviceSynchronize 接口，获取对应的 assert 错误码。

**使用方法**

- assert 接口是 SDAA C 提供的内置接口，使用时无需引入头文件。
- assert 接口的参数与 C/C++ 提供的 assert 接口和用法相同。

**使用示例**

以检测除数不为 0 作为示例，assert 会检查除数是否为 0，为 0 时会打印错误所在的文件行号信息并且终止 SPE 后续程序的执行。

```c
__device__ int g_sum = 0;
__device__ void my_div(int a, int b)
{
    // 如果除数不为0，则打印错误信息并终止SPE后续程序的执行
    assert(b != 0 && "b should not be zero");
    g_sum = a / b;
}


__global__ void bar()
{
    my_div(1, 2);
}


int main()
{
    sdaaSetDevice(0);
    bar<<<1>>>();
    sdaaDeviceSynchronize();


    return 0;
}
```

### 13.2 性能度量

#### 13.2.1 性能指标

本文主要介绍衡量程序性能的关键指标：执行周期和访存带宽。

**执行周期**

在 SDAA C 中，使用执行周期作为衡量程序性能的关键指标。

程序性能公式如下：

```
性能 = (1/程序) ⋅ (指令/程序) ⋅ (执行周期/指令) ⋅ (时间/执行周期)
     = (1/程序) ⋅ (执行周期/程序) ⋅ 时钟周期
```

其中：

- 时钟周期（单位：秒）：计算核心 SPE 基本的时间单位，指发生一个最小事件所需的时间，由计算核心 SPE 的时钟频率决定。
  - 时钟周期的公式：`时钟周期 = 1 / (SPE 的时钟频率)`
- 执行周期（单位：拍）：程序运行所消耗的时钟周期个数（也称作拍数，一拍对应一个时钟周期）。

由于时钟周期是由计算核心 SPE 的时钟频率所决定的固定值，那么执行周期越短，程序的运行效率越高效，程序性能越好。因此，执行周期是衡量程序性能的关键指标。

> 说明：
> - 非访存类指令的执行周期通常是固定的。
> - 访存类指令的执行周期与数据量、访存方式以及时钟频率有关。

**访存带宽**

对于以数据读写为主的程序，访存带宽（即 IO 带宽）是通用的性能指标。

访存带宽是计算核心 SPE 在单位时间内读取或写入内存的数据量，通常以每秒传输的比特数或字节数进行衡量。

*理论带宽*

理论带宽的计算公式如下：

```
理论带宽（Byte/s） = 内存最大频率 × 内存通道数 × 648（每次传输位宽，单位：bit）
```

*访存带宽*

在实际使用中，由于算法效率、内存延迟等均会影响数据传输的效率，因此访存带宽通常低于理论带宽。

访存带宽的计算公式如下：

```
访存带宽（Byte/s） = 数据传输量（Byte）× 频率（Hz）/ 执行周期（拍）
```

由以上公式可以看出：执行周期越短，访存带宽越大。

#### 13.2.2 性能采样

使用 perf_start、perf_stop 和 perf_print 函数可以对 SDAA C 程序进行性能采样，采集目标代码段的执行周期。目前，性能采样工具只输出两个主要的性能数据，一个是 perf_start、perf_stop 之间的执行周期（延时拍数），另一个是指令缓存脱靶次数（Instruction Cache Miss）。

> 说明：
> 使用性能采样函数前，需要手动包含 sdaa_perf.h 头文件。

- perf_start：开始性能采样接口，通过此接口标识性能采样的起始位置。
- perf_stop：结束性能采样接口，通过此接口标识性能采样的结束位置。
- perf_print：在终端界面输出采集到的性能数据（延时拍数和指令缓存脱靶次数）。

**性能采样示例**

> 说明：
> 为准确测量性能，可以循环多次调用被测程序并计算执行周期的平均值，以减少缓存未命中（Cache Miss）所带来的性能偏差。

使用 perf_start 和 perf_stop 进行性能采样的示例如下：

```c

#define NUM 51200
#define FSIZE (NUM * sizeof(float))


__global__ void kernel_sig(float *a, uint32_t num)
{

        perf_start();

             a[i] = sigmoidf(a[i]);

{
```

#### 13.3.1 程序并行

##### 13.3.1.1 任务级并行

本文主要介绍任务级并行的性能优化策略和示例。

**优化策略**

太初 AI 加速卡目前提供多个计算核心阵列 SPA，每个 SPA 构成一个独立的进程。任务级并行主要指：同时并行运行多个计算核心阵列 SPA 完成相应的计算任务。

**优化示例**

| v1.2.0 | 2.36GHz |

**优化前**

只使用一个 SPA 计算 40000 个数的 sigmoid 输出值。

```c
#include <sdaa_perf.h>

#define DEV_NUM 4
#define NUM 10000
#define FSIZE (NUM * sizeof(float))
```

**优化后**

将 40000 个数改为进程间共享数据，本文以 4 等份为例，同时并行运行 4 个 SPA 计算 40000 个数的 sigmoid 输出值，每个 SPA 计算一份数据。

```c
#include <sdaa_perf.h>
#include <fcntl.h>
#include <iostream>
#include <string>
#include <sys/mman.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>
using namespace sdaa;


#define DEV_NUM 4
#define NUM 10000
#define FSIZE (NUM * sizeof(float))


__global__ void kernel_sig(float *a, uint32_t num)
{
    if(threadIdx == 0){
        perf_start();
        for (uint32_t i = 0; i < num; i++) {
             *(a + i) = sigmoidf(*(a + i));
        }
        perf_stop();
        perf_print();
    }
}

{
    float *shared_mem;
    // 创建共享内存
    int fd = shm_open("/my_shared_memory", O_CREAT | O_RDWR, S_IRUSR | S_IWUSR);

    // 共享内存 大小分成 4 份
    shared_mem = (float *)mmap(NULL, (DEV_NUM * FSIZE), PROT_READ | PROT_WRITE,
                                MAP_SHARED, fd, 0);


    for (int i = 0; i < (DEV_NUM * NUM); i++) {
        shared_mem[i] = (float)i / NUM;
    }
    // 创建第一个子进程
    pid_t pid1 = fork();
    if (pid1 == 0) {
        // 第一个子进程中使用第一个计算核心阵列SPA
        sdaaSetDevice(0);

        // 运行kernel 函数 进行计算
        kernel_sig<<<1>>>(dev_input, NUM);
        sdaaDeviceSynchronize();
        // 将计算过的 device 上的数据复制到第一个1/4的共享内存
        sdaaMemcpy(shared_mem, dev_input, FSIZE, sdaaMemcpyDeviceToHost);
        sdaaFree(dev_input);
        exit(0);
    }


    // 创建第二个子进程
    pid_t pid2 = fork();
    if (pid2 == 0) {
        // 第二个子进程中使用第二个计算核心阵列SPA
        sdaaSetDevice(1);
        // 在Device端申请内存，大小是 共享内存的 1/4
        float *dev_input = NULL;
        sdaaMalloc((void **)&dev_input, FSIZE);
        // 将共享内存的第二个1/4数据复制到 第二个device 的内存上
        sdaaMemcpy(dev_input, (shared_mem + NUM), FSIZE,
                   sdaaMemcpyHostToDevice);
        // 运行kernel 函数 进行计算
        kernel_sig<<<1>>>(dev_input, NUM);
        sdaaDeviceSynchronize();
        // 将计算过的 device 上的数据复制到第二个1/4的共享内存
        sdaaMemcpy((shared_mem + NUM), dev_input, FSIZE,
                   sdaaMemcpyDeviceToHost);
        sdaaFree(dev_input);
        exit(0);
    }


    // 创建第三个子进程
    pid_t pid3 = fork();
    if (pid3 == 0) {
        // 第三个子进程中使用第三个计算核心阵列SPA
        sdaaSetDevice(2);
        // 在Device端申请内存，大小是 共享内存的 1/4
        float *dev_input = NULL;
        sdaaMalloc((void **)&dev_input, FSIZE);
        // 将共享内存的第三个1/4数据复制到 第三个device 的内存上
        sdaaMemcpy(dev_input, (shared_mem + 2 * NUM), FSIZE,
                   sdaaMemcpyHostToDevice);
        // 运行kernel 函数 进行计算
        kernel_sig<<<1>>>(dev_input, NUM);
        sdaaDeviceSynchronize();
        // 将计算过的 device 上的数据复制到第三个1/4的共享内存
        sdaaMemcpy((shared_mem + 2 * NUM), dev_input, FSIZE,
                   sdaaMemcpyDeviceToHost);
        sdaaFree(dev_input);
        exit(0);
    }


    // 创建第四个子进程
    pid_t pid4 = fork();
    if (pid4 == 0) {
        // 第四个子进程中使用第四个计算核心阵列SPA
        sdaaSetDevice(3);
        // 在Device端申请内存，大小是 共享内存的 1/4

        sdaaDeviceSynchronize();
        // 将计算过的 device 上的数据复制到第四个1/4的共享内存

                   sdaaMemcpyDeviceToHost);
        sdaaFree(dev_input);
        exit(0);
    }


    pid_t pid;
    int status = 0;
    // 等待子进程全部执行完成
    while ((pid = wait(&status)) > 0)
        ;


    // remove share memory
    munmap(shared_mem, (4 * FSIZE));
    shm_unlink("/my_shared_memory");


    return 0;
}
```

##### 13.3.1.3 指令级并行（SIMD）

本文主要介绍指令级并行（SIMD）的性能优化策略和示例。

**优化策略**

太初 AI 加速卡的每个计算核心 SPE 都有多个向量寄存器，运用 SIMD（Single Instruction Multiple Data，单指令流多数据流）向量指令的加速能力，不仅可以降低功耗，还可以提高性能，满足系统需要。

**优化示例**

计算 51200 次浮点加法。

**测试环境**

| TecoDriver 和 TecoToolKit 版本 | 计算核心频率 |
|------|------|
| v1.2.0 | 2.36GHz |

**优化前**

使用 SPMD 编程模型，一个 SPA 内所有 SPE 分别执行 1600 次浮点标量计算，实现 51200 次浮点加法。

```c
#include <sdaa_perf.h>
using namespace sdaa;


#define NUM 51200
#define FSIZE (NUM * sizeof(float))


__global__ void kernel_add(float *a, uint32_t num)
{
        perf_start();
        // 每次循环计算是跨步为 threadDim，为每个SPE分配一个数据去计算
        for (uint32_t i = 0; i < num; i += threadDim) {
            // 用 i+threadIdx 作为坐标，每个SPE都参与计算
            a[i + threadIdx] = a[i + threadIdx] + 10;
        }
        perf_stop();
        perf_print();
}


int main()
{
        sdaaSetDevice(0);
        // 主机端申请数据空间
        float *host_data = (float *)malloc(FSIZE);
        for (int i = 0; i < NUM; i++) {
            host_data[i] = (float)i / NUM;
        }
        float *dev_input = NULL;
        // 申请Device端数据空间
        sdaaMalloc((void **)&dev_input, FSIZE);
        // 将主机端数据复制到Device端
        sdaaMemcpy(dev_input, host_data, FSIZE, sdaaMemcpyHostToDevice);
        // 核函数运行计算
        kernel_add<<<1>>>(dev_input, NUM);
        sdaaDeviceSynchronize();
        // 将计算好的数据从Device端复制到主机端
        sdaaMemcpy(host_data, dev_input, FSIZE, sdaaMemcpyDeviceToHost);


        free(host_data);
        sdaaFree(dev_input);
        return 0;
```

**优化后**

使用 SPMD 编程模型和 SIMD 优化，一个 SPA 内所有 SPE 分别执行 100 次浮点向量计算，实现 51200 次浮点加法。

```c
#include <sdaa_perf.h>
using namespace sdaa;


#define NUM 51200
#define FSIZE (NUM * sizeof(float))
#define SIMD_NUM 16


__global__ void kernel_add(float *a, uint32_t num)
{
    perf_start();
    floatv16 va;
    // 每次循环计算是跨步为 threadDim*16，为每个SPE分配16个数据去计算
    for (uint32_t i = 0; i < num; i += (threadDim * SIMD_NUM)) {
        // 用 i+threadIdx*SIMD_NUM
        // 作为坐标，每个SPE都往向量寄存器装入16个float数
        simd_load(va, (a + i + threadIdx * SIMD_NUM));
        // 每个SPE 都做 add的运算
        va = va + 10;
        // 将 每个SPE的向量寄存器里的16个float数据， 存回到对应地址
        simd_store(va, (a + i + threadIdx * SIMD_NUM));
    }
    perf_stop();
    perf_print();
}


int main()
{
    sdaaSetDevice(0);
    // 主机端申请数据空间
    float *host_data = (float *)malloc(FSIZE);
    for (int i = 0; i < NUM; i++) {
        host_data[i] = (float)i / NUM;
    }
    // simd_load/simd_store 需要做数据地址的 64Byte对齐
    float __attribute__((aligned(64))) *dev_input = NULL;
    // 申请Device端数据空间
    sdaaMalloc((void **)&dev_input, FSIZE);
    // 将主机端数据复制到Device端
    sdaaMemcpy(dev_input, host_data, FSIZE, sdaaMemcpyHostToDevice);
    // 核函数运行计算
    kernel_add<<<1>>>(dev_input, NUM);
    sdaaDeviceSynchronize();
    // 将计算好的数据从Device端复制到主机端
    sdaaMemcpy(host_data, dev_input, FSIZE, sdaaMemcpyDeviceToHost);


    free(host_data);
    sdaaFree(dev_input);
    return 0;
}
```

#### 13.3.2 函数接口

##### 13.3.2.1 SPE 同步接口

本文主要介绍使用线程组同步 SPE 接口的性能优化策略和示例。

**优化策略**

使用线程组同步 SPE 接口，加入线程组的 SPE 在满足以下条件时，性能较好：

SPE 的 ID 号除以 8 所得的商或者余数相同。

例如：使用包括 4 个 SPE 的线程组进行同步时，线程组 A={0, 1, 2, 3} 或 B={0, 8, 16, 24} 的性能要优于线程组 C={0, 9, 18, 27}。

**优化示例**

##### 13.3.2.2 DMA 数据搬运

###### 13.3.2.2.1 SPE ID 相关

本文主要介绍使用不同线程组进行 DMA 数据搬运的性能优化策略和示例。

**优化策略**

使用 DMA 数据搬运接口时，在数据搬运量一定的情况下，加入线程组的 SPE 满足以下条件时，性能较好：

- SPE 的 ID 号除以 8 所得的余数尽可能不相同。
- 在无法避免需使用有相同余数的 SPE 情况下，启用的有相同余数的 SPE 的数量尽可能小。

例如：使用 8 个 SPE 的线程组进行数据搬运时：

- 线程组 A={0, 1, 2, 3, 4, 5, 6, 7} 的性能要优于线程组 B={0, 1, 2, 3, 8, 9, 10, 11}。
- 线程组 B={0, 1, 2, 3, 8, 9, 10, 11} 的性能要优于线程组 C={0, 1, 8, 9, 16, 17, 24, 25}。

> 提示：
> 该优化策略适用于非阻塞型 DMA 数据搬运及阻塞型 DMA 数据搬运中非 SPM 存储空间之间的数据搬运。

**优化示例**

使用 8 个 SPE 的线程组进行非阻塞型 DMA 数据搬运时，测试以下几种线程组的执行周期并比较：

- A={0, 1, 2, 3, 4, 5, 6, 7}，余数均不相同，即相同余数的 SPE 数量最大值为 1。
- B={0, 1, 2, 3, 8, 9, 10, 11}，有 2 组相同的余数（0, 1, 2, 3），相同余数的 SPE 数量最大值为 2。
- C={0, 1, 8, 9, 16, 17, 24, 25}，有 4 组相同的余数（0, 1），相同余数的 SPE 数量最大值为 4。

每个测例均调用 10000 次 memcpy_async（详情请参考《SDAA C 编程指南》的 memcpy_async 老接口章节）计算平均值以消除初次调用时 cache 影响。
