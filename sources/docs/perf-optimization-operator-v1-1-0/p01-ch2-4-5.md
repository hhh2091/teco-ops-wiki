## 3. 性能度量

**性能指标**

本文对性能的定义主要从工程的角度出发，主要关注程序在处理器上运行的直接性能表现，即程序的执行速度，包含两方面：

- 以浮点运算为主的算法，采用浮点性能 FLOPS（每秒执行的浮点运算数）作为关键指标。
- 以数据读写为主的算法，采用 IO 带宽（通常以 GB/s 为单位）作为关键指标。

**计算方法**

为了实际测算具体算法的性能指标，最常用的方法是测量算法在计算给定问题上的耗时，然后根据计算过程所需的浮点运算数 FLOPs 或数据读写量来计算浮点性能与 IO 带宽。性能采样工具的使用，可以参考 SDAA C 编程指南的性能采样（详情请参考《SDAA C 编程指南》的性能采样章节）章节。

计算公式：

- 浮点性能（FLOPS）= 浮点运算次数 / 运算时间（秒）
- IO 带宽（GB/s）= 传输的数据量（字节） / 传输时间（秒）

> **说明：**
>
> 针对设备端的性能优化，暂不考虑主机端执行、主机-设备之间数据传输的开销，只需测量设备端 Kernel 函数或其中关键代码段的执行用时。

## 4. 性能优化

### 4.1 概述

并行算法优化过程并不保证浮点性能与 IO 带宽两种指标同时接近硬件的峰值，这是由特定算法的计算密度决定的。针对特定算法进行优化，可以根据算法的计算密度，选择相应的优化方式。

**计算密度**

计算密度，也称为计算访存比，指的是一个算法在执行过程中，进行的浮点运算次数与读写数据量的比值，单位：FLOPs/Byte。计算密度越大，说明算法越集中于浮点计算，而非数据读写。

例如，对于一个输入张量形状为 NHWC，使用形状为 3x3 的 2D 卷积核，输出张量形状为 NPQK 的卷积运算，假设所有数据类型均为 float，其计算密度 I 为：

$$
I = \frac{NPQK * (2 * 3 * 3 - 1) * C}{(NPQK + NHWC + C * 3 * 3 * K) * \mathrm{sizeof}(\mathrm{float})}
$$

假设理想情况下，算法的浮点性能和 IO 带宽都能达到硬件的峰值，分别记为 BW_comp 和 BW_IO，那么算法用于浮点运算和数据读写的时间之比即为：

$$
I * \frac{BW_{IO}}{BW_{comp}}
$$

**算法类别**

对于特定硬件，峰值 IO 带宽与浮点性能的比值是确定的。因此，算法用于浮点运算和数据读写的时间之比完全取决于计算密度。据此，可以将不同的算法进行分类：

- 运算时间远大于数据读写时间，称为计算密集型算法。
- 运算时间远小于数据读写时间，称为访存密集型算法。
- 运算与数据读写时间相当，称为均衡型算法。

其中：

- 对于密集型算法，其执行速度由对应行为的性能指标决定。因此对其展开优化，应针对相应的性能指标，使其尽量接近硬件的峰值性能。
- 对于均衡型算法，其整体性能同时受计算和访存两方面影响，两方面的性能指标优化都能带来明显的收益。同时，如果能利用硬件的特性，设法让运算与数据读写过程并发执行，可以进一步降低算法的执行用时，提升整体性能。

后续章节将分别介绍计算和访存两方面的优化手段，并给出相应的优化实践案例。基于这些优化方法，用户可以结合自己的需求，既可用于相应的密集型算法的优化，也可同时运用于均衡型算法的性能提升。

### 4.2 计算相关优化

并行程序设计的首要原则是尽量将任务拆分成可以并行处理的子任务，分配给处理器的多个计算单元并行处理。

每个计算核心在执行运算时，应当充分利用现代向量处理器的向量计算能力和流水线设计，尽可能实现数据并行和指令并行，才能发挥硬件的峰值性能。

此外，除了通用运算和控制功能外，某些处理器会提供针对特殊运算的硬件，在合适的场景下，能够获得明显优于通用运算的性能表现。

本章将对这三方面依次进行介绍：

- 向量指令使用
- 指令流水线优化
- 矩阵乘法加速单元

#### 4.2.1 向量指令使用

**优化策略**

单指令多数据（SIMD）是一种典型的数据并行方法，指的是通过一条指令同时对多个不同的数据进行运算的设计思想。在现代向量处理器中，SIMD 就是由各种向量指令支持的。

例如，普通的一条标量加法指令只能完成一个加数与一个被加数的求和运算。而一条向量加法指令则可以一次完成多个加数与等量的被加数的求和。虽然两种指令的延迟可能存在一定差异，但由于向量指令能一次完成更多的运算，其整体运算速度是明显高于标量运算的。

因此，在算法设计以及性能调优的过程中，应当尽可能采用 SIMD 的思路，广泛使用向量指令进行加速。

需注意，SIMD 中并行的数据，需要相互不存在数据依赖。因此，使用 SIMD 时，需要设法解除原本的依赖关系。在加法示例中，对 N 个输入整数求和，顺序求和算法，意味着每次加法依赖于之前求和的结果，所有数据间产生了依赖关系，因此无法进行数据并行。但是利用加法的结合律，将 N 个输入均分为若干组，每一组可以无依赖地分别求和，最终对各组的结果求和得到输出。这样就实现了各组之间的数据并行，可以使用向量指令进行加速。

因此，在算法设计层面，需要开发者充分理解 SIMD 的思想，主动消除数据依赖关系，才能广泛地应用向量指令实现加速。在太初 AI 加速卡上支持的向量指令的功能和使用方法可以参考 SDAA C 编程指南中的向量操作（详情请参考《SDAA C 编程指南》的向量操作章节）章节。

**优化示例**

本节以一元定积分为例，介绍如何解除数据依赖，使用向量指令加速浮点数运算过程。

**普通实现**

给定一个一元函数 f(x) 和一个闭区间 [a, b]，计算该区间上的定积分 I = ∫ₐᵇ f(x)dx，可以通过矩形公式（0 阶近似）离散计算 I 的近似值，将积分区间均分为 N 等分：

$$
I = \sum_{i=1}^{N} f(x_i)\,\triangle
$$

其中：

$$
\triangle = (b - a)/N
$$

$$
x_i = a + (i - 1)\triangle
$$

当 N 足够大时，离散积分的结果可以作为定积分精确值的高度近似。因此，将积分区间拆分到各个计算核心上，每个核心负责一个子区间的积分，最终由一个核心（假设为 SPE0）将各子区间的积分结果求和，即可输出最终的积分值。伪代码实现如下：

```c
// Calculate the width of each subinterval
float delta = (b - a) / N;


// Determine the starting point of the subinterval for the current thread
float x_beg = a + delta * threadIdx * N / threadDim;


// Determine the ending point of the subinterval for the current thread
float x_end = a + delta * (1 + threadIdx) * N / threadDim;


// Compute the integral over the subinterval assigned to the current thread
float sub_integral = integrate(x_beg, x_end, N / threadDim);


// Synchronize all threads to ensure all sub_integrals are computed
sync_threads();


if (0 == threadIdx) {
    // If this is the first thread (threadIdx == 0),
    // gather sub_integral values from all other threads using RMA (Remote Memory Access)
    RMA get sub_integral from all other threads;


    // Sum up all the sub_integral values to get the total integral
    float sum = sum all sub_integral;


    // Write back the final result of the integration
    write back sum;
}
```

其中，单个子区间的积分是通过函数 `integrate()` 的矩形公式实现的。

以 f(x) = x²，[a, b] = [−3, 3]，N = 10⁶ 为输入参数，使用上述并行策略，在 SPE 主频为 2.36GHz 的条件下，使用 SDAA C 编程指南的性能采样功能（详情请参考《SDAA C 编程指南》的性能采样章节）计算 Kernel 函数中并行积分计算代码的执行用时（不包括 RMA 通讯用时），平均约为 107us。

**优化实现**

基于上述并行版本，可以进一步优化，即在负责每个计算核心的子区间积分过程中，进行数据并行：将子区间内所有参与离散积分的点的积分过程拆分为 16 组，即：

$$
I = \sum_{i=1}^{16} \sum_{j=1}^{N/16} f(x_i + 16(j - 1)\triangle)\,\triangle
$$

这 16 组积分过程是独立的，只需要最终将分别积分的结果累加即可。这样就构造了 16 组不存在数据依赖的计算过程，可以用太初 AI 加速卡的单精度浮点数向量指令来同时运算。因此，`integrate()` 函数的实现如下：

```c
__device__ float integrate_v1(const float a, const float b, const int N) {
    // discrete interval
    const float dx = (b - a) / N;
    // compute integral, using 0-th order approximation
    float integral = 0.0f;
    floatv16 vx = a;
    floatv16 vdx = dx;
    {
        floatv16 offset =
              {0.0f, 1.0f, 2.0f, 3.0f, 4.0f, 5.0f, 6.0f, 7.0f, 8.0f, 9.0f, 10.0f, 11.0f, 12.0f, 13.0f, 14.0f,
        vx += offset * vdx;
    }
    floatv16 vres = 0.0f;


    // loop and use SIMD
    int i;
    for (i = 0; i <= N - 16; i += 16) {
        vres += vx * vx;
        vx += vdx;
    }


    // loop the rest part
    float x = a + i * dx;
    while (i < N) {
        integral += x * x;
        x += dx;
        i++;
    }


    // reduce result
    float res[16];
    simd_store(vres, res);
    for (i = 0; i < 16; i++) {
        integral += res[i];
    }
    integral *= dx;
    // over
    return integral;
}
```

**优化效果**

采用上述向量指令优化后，在相同条件下测得的 Kernel 执行用时，降低到平均 8us，也就是获得了约 13 倍的加速比。

#### 4.2.2 指令流水线优化

**优化策略**

处理器的每个硬件核心通常有多个指令流水线，是实现多指令（MI）并行设计思想的基础。

所谓指令流水线，是将一条指令在处理器上执行的过程，分解为多个串行的阶段，而非一个不可拆分的整体，指令（及其数据）随着时钟信号依次通过所有阶段。通过流水线化，处理器可以并行处理多个指令，仅轻微增加每条指令的延迟。

例如，处理器连续处理 3 条相同功能的指令：

- 图 1 是非流水线化的执行过程：每条指令必须等待前一条完全执行完成才能得到执行，总的延迟是每条指令延迟的 3 倍。

  > 图 1 非流水线化指令处理

- 图 2 是三阶段流水线化的执行过程，每条指令只需要等待前一条指令的第一阶段执行完毕就能开始执行，总的延迟明显低于非流水线的情况。

  > 图 2 流水线化指令处理

指令流水线通过增加指令执行的阶段划分，提升了处理器的指令吞吐量（单位时间内能执行的指令条数）。此外，现代处理器广泛采用多发射流水线技术：在流水线的每个阶段，可以同时处理不止一条指令，这同样可以大幅度提升处理器吞吐量。

因此，为了充分利用处理器流水线提供的指令吞吐量，开发者应当在同一时间提供足够数量的相同指令（在寄存器等资源足够的前提下），由处理器并行地执行这些指令，将指令流水线的各个阶段尽量占满。

**优化示例**

在上节使用向量指令优化后的算法中，计算子区间积分的循环每次只有一条浮点向量乘加指令在执行。而太初 AI 加速卡的每个计算核心都支持多级指令流水线，当多条无依赖的指令同时执行时，流水线才能被这些指令填满，达到最佳运算性能。因此，可以尝试进一步的优化：在循环内同时执行多条浮点向量乘加指令，也就是循环展开。示例选择将循环展开两次，即每轮循环内同时执行 2 条乘加指令，相应的 Kernel 代码如下所示：

```c
__device__ float integrate_v2(const float a, const float b, const int N) {
   // discrete interval
   const float dx = (b - a) / N;
   // compute integral, using 0-th order approximation
   float integral = 0.0f;
   floatv16 vx[2], vres[2], vdx = dx;


     vres[0] = vres[1] = 0.0f;
    {
        vx[0] = a;
        vx[1] = a;
        floatv16 offset0 =
              {0.0f, 1.0f, 2.0f, 3.0f, 4.0f, 5.0f, 6.0f, 7.0f, 8.0f, 9.0f, 10.0f, 11.0f, 12.0f, 13.0f, 14.0f,
        floatv16 offset1 = offset0 + (floatv16)16.0f;
        vx[0] += offset0 * vdx;
        vx[1] += offset1 * vdx;
    }


    // loop and use SIMD
    int i;
    for (i = 0; i <= N - 32; i += 32) {
        vres[0] += vx[0] * vx[0];
        vres[1] += vx[1] * vx[1];


        vx[0] += vdx;
        vx[1] += vdx;
    }


    // loop the rest part
    float x = a + i * dx;
    while (i < N) {
        integral += x * x;
        x += dx;
        i++;
    }


    // reduce result
    vres[0] += vres[1];
    float res[16];
    simd_store(vres[0], res);
    for (i = 0; i < 16; i++) {
        integral += res[i];
    }
    integral *= dx;
    // over
    return integral;
}
```

**优化效果**

在同样的场景下测试，循环展开后的积分过程，耗时可以降低约 25%。

#### 4.2.3 矩阵乘法加速单元

**优化策略**

不同处理器硬件由于面向的应用场景不同，往往会提供一些针对特定问题（例如特殊函数、矩阵乘法等）加速的计算硬件，被称为特殊计算单元（Special Function Unit，或 SFU）。SFU 虽然只能用于特定计算，但由于其加速基于硬件实现，而非处理器通用指令的软件实现，其计算速度远远高于通用运算过程。因此，用户可以尽可能增加 SFU 的利用，而非通用指令。

在太初 AI 加速卡中，最重要的 SFU 是矩阵乘法单元，专门用于进行矩阵的乘加运算（Matrix Multiply-Accumulate，或 MMA）。通过矩阵乘法单元实现的 MMA 计算，比依靠处理器的通用乘加指令要快得多。在需要大量计算 MMA 的算法中，例如 GEMM、卷积等算子，合理使用矩阵乘法单元，可以获得很高的浮点运算性能。具体的使用方法可以参考 SDAA C 编程指南的矩阵乘（详情请参考《SDAA C 编程指南》的矩阵乘章节）章节。

**优化示例**

本节示例将展示如何使用矩阵乘法单元，在 SPM 上计算两个矩阵的乘法。

示例中的输入输出数据类型均为 half 类型，左矩阵形为 128×128，右矩阵形为 128×32。由于矩阵乘接口每次进行的是一个 128×32×32 的 MMA，因此，需要预先将左矩阵拆分为若干 128×32 的分块存储，方便矩阵乘接口的加载。相应的 Kernel 代码如下所示：

```cpp
__global__ void mma_using_ace(void) {
    half* left_mat = (half*)malloc(sizeof(half) * 128 * (32 * 4));
    half* right_mat = (half*)malloc(sizeof(half) * 32 * (32 * 4));
    half* output = (half*)malloc(sizeof(half) * 128 * 32);


    MatmulHandle handle;
    matmul_init(handle, MatmulHalfToHalf);
    matmul_set_flushing_output(handle, false);


    // the 1st three rounds
    for (int i = 0; i < 3; i++) {
        matmul_load_weight(handle, right_mat + i * 1024, MatmulK32, MatmulN32);
        matmul_compute(handle, left_mat + i * 128 * 32, 128, MatmulK32);
    }
    // the last round
    matmul_set_flushing_output(handle, true);
    matmul_load_weight(handle, right_mat + 3 * 1024, MatmulK32, MatmulN32);
    matmul_compute(handle, left_mat + 3 * 128 * 32, 128, MatmulK32);


    matmul_store(handle, output, 128, MatmulN32);


    free(left_mat);
    free(right_mat);
    free(output);
    // over
    return;
}
```

**优化效果**

在 SPE 主频为 2.36GHz 的条件下，测得其浮点性能约为 29 TFLOPS，而通用运算指令的浮点性能峰值约为 2.5 TFLOPS。因此，在条件允许的情况下，应当尽量充分利用矩阵乘法单元进行矩阵乘计算。

需要指出的是，上述实验远没有达到太初 AI 加速卡乘加运算的浮点性能峰值，在合适的应用场景下，充分利用矩阵乘法单元可以获得更高的浮点计算性能。

### 4.3 访存相关优化

太初 AI 加速卡设备端的存储器模型可以分为 3 层：Global 存储、SPM 存储和寄存器。

- Global 存储是所有片上所有核心共享的大容量存储器。
- SPM 存储是每个计算核心的本地高速缓存，属于计算核心的私有资源。
- 寄存器是核心执行运算和控制时，使用的特殊存储资源。

从访问速度而言，寄存器 > SPM 存储 > Global 存储。从存储容量而言，Global 存储 > SPM 存储 > 寄存器。

而一个算法运行过程的数据流动通常包括如下过程：

1. 从主机端将输入数据传输到 Global 存储。
2. 计算核心从 Global 存储读取数据到自己的 SPM 中。
3. 计算核心使用 SPM 中的数据进行计算，这个过程会根据指令执行的需要，使用各种寄存器，各种中间结果和阶段性输出都暂存在 SPM 中。
4. 计算完成后，SPM 上的输出结果写回 Global 存储。
5. 主机端从 Global 存储中取回输出结果。

> 图 1 数据流向示意图

其中，

- 阶段 1 和 5，主机端与 Global 存储之间的数据交换可以参考 SDAA C 编程指南中的数据传输（详情请参考《SDAA C 编程指南》的数据传输章节）章节进行调用和优化。
- 阶段 3，计算过程中访问 SPM 以及使用寄存器的过程，通常不需要特殊的控制。
- 阶段 2 和 4，由于 Global 存储访问速度最慢，容量最大，输入输出阶段都需要对其进行大量的数据读写，因此是需要集中优化的阶段。

Global 存储的具有单次访问延迟很高，但吞吐量（即单次访问的数据量）很大的特点。因此，它不适合像 SPM 或寄存器一样，以高频多次少量的方式进行读写。Global 存储最佳的读写模式是一次读写大量数据，控制访问频次。因此，可以考虑尽可能实现：

- 合并读取：将多次计算所需的输入数据作为一个整体，通过一次对 Global 存储的访问操作，读入 SPM。
- 合并写出：使用 SPM 上的数据进行计算，输出结果暂存在 SPM 上，将 SPM 上的结果整体写回 Global 存储。

#### 4.3.1 双缓冲优化

**优化策略**

主机与设备间的数据传输可以与设备上 Kernel 的执行并行，互不影响。这意味着：主机端可以通过异步的数据传输接口发起传输，然后立即发起 Kernel 的执行操作，实现访存和计算的并行，进一步缩短程序耗时。同样的，使用 Global 存储与 SPM 间的非阻塞型 DMA（详情请参考《SDAA C 编程指南》的非阻塞型 DMA 数据搬运章节）接口，可以实现访存和计算的并行。

这意味着开发者需要解除并行数据传输与计算过程的数据依赖，也就是计算过程不能依赖于同一时刻正在传输的数据。为了实现这一点，开发者需要在 SPM 上设置至少 2 个数据缓冲区。

- 计算缓冲区：存储计算过程依赖的数据。
- 传输缓冲区：存储异步传输的数据。

当计算和访存都完成后，再交换两个缓冲区的职能，就能始终确保计算过程与数据传输过程无依赖。

正是因为这种同时需要 2 个数据缓冲区的特点，此优化方法被称为双缓冲优化。

**优化示例**

本节以 1D 卷积算法为例，演示如何使用双缓冲技术优化 Global 存储的数据访问。即输入一个长度为 N 的一维信号（单通道），使用一个长度为 3 的卷积核对其进行滤波，输出长度仍为 N 的滤波结果。

**普通实现**

算法的基本设计思路：

1. 首先，需要根据 SPM 大小，确定每次最多能计算输出信号的长度 dN。据此将计算过程分解为若干部分，由不同的 SPE 并行。
2. 在每个 SPE 上，给定需要计算的输出信号范围后，先从 Global 存储中将计算过程所依赖的输入信号读入 SPM，然后进行计算，并将结果从 SPM 的输出缓冲区写回 Global 存储的对应位置。

以伪代码描述各个 SPE 的并行计算过程如下：

```c
for (i = threadIdx; i < N / dN; i += threadDim) {
      Determine the output signal range that needs to be computed: y[i*dN, (i+1)*dN)
      Determine the input signal range needed for computation: x[i*dN-1, (i+1)*dN]
      Read the input signal range x[i*dN-1, (i+1)*dN] from Global storage into the input buffer spm_x in
SPM
      Perform the convolution filtering on spm_x and store the result in the output buffer spm_y in SPM
      Write the spm_y output buffer back to Global storage
}
```

在 stride = 1，padding = 1，预留位置填充 0 的情况下，一维卷积（3 点平滑）的基础版本 Kernel 函数如下：

```c
__global__ void test_3pt_smooth_baseline(float* x, const int N, float* y) {
      constexpr int max_tile_size = 8192;


      // parallelize
      const int tile_num = (N + max_tile_size - 1) / max_tile_size;
      const int last_tile_size = N - (tile_num - 1) * max_tile_size;


    // allocate SPM
    float* tile_x = (float*)malloc(sizeof(float) * (2 + max_tile_size));
    float* tile_y = (float*)malloc(sizeof(float) * max_tile_size);


    // traverse all tiles
    for (int i = threadIdx; i < tile_num; i += threadDim) {
         const int tile_size = (i < tile_num - 1) ? max_tile_size : last_tile_size;


         // read tile of x
         memcpy(tile_x, x + i * tile_size - 1, sizeof(float) * (tile_size + 1));


         // perform 3-point smoothing
         if (i == 0) {
              tile_x[0] = 0.0f;
         }
         if (i == tile_num - 1) {
              tile_x[tile_size] = 0.0f;
         }
         perform_3pt_smooth(tile_x, tile_size, tile_y);


         // output tile of y
         memcpy(y + i * tile_size, tile_y, sizeof(float) * tile_size);
    }


    // free SPM
    free(tile_x);
    free(tile_y);


    // over
    return;
}
```

**优化实现**

对上述基础版本进行双缓冲优化，伪代码如下：

```c
Determine the first segment of the output signal range that the current SPE needs to compute:
y[threadIdx*dN, (threadIdx+1)*dN)
Determine the corresponding input signal range: x[threadIdx*dN-1, (threadIdx+1)*dN]
Set the buffer flag for computation to flag = 0
Use an asynchronous interface to read x[threadIdx*dN-1, (threadIdx+1)*dN] from Global storage into one
of the double buffers spm_x[flag]
for (i = threadIdx; i < N / dN; i++) {
    Wait for the previously initiated data transfer to complete
    Determine the next segment of the output signal range to compute: y[(i+threadDim)*dN,
(i+threadDim+1)*dN)
    Determine the corresponding input signal range: x[(i+threadDim)*dN-1, (i+threadDim+1)*dN)
    Use an asynchronous API to read the next segment of the input signal into the other buffer
spm_x[flag ^ 1]
    Perform the convolution filtering on spm_x[flag], and write the result into the output buffer spm_y
in SPM
    Write spm_y back to Global storage
    Swap the buffer flag, i.e., flag = flag ^ 1
}
```

双缓冲版本的 Kernel 函数如下：

```c
__global__ void test_3pt_smooth_v1(float* x, const int N, float* y) {
   constexpr int max_tile_size = 8192;


   // parallelize
   const int tile_num = (N + max_tile_size - 1) / max_tile_size;
   const int last_tile_size = N - (tile_num - 1) * max_tile_size;


   // allocate SPM
   int flag = 0;
   float* tile_x[2];
   tile_x[0] = (float*)malloc(sizeof(float) * (2 + max_tile_size));
   tile_x[1] = (float*)malloc(sizeof(float) * (2 + max_tile_size));


   float* tile_y = (float*)malloc(sizeof(float) * max_tile_size);


   if (threadIdx < tile_num) {
        const int tile_size = (threadIdx < tile_num - 1) ? max_tile_size : last_tile_size;
        // read tile of x
        memcpy_async(tile_x[flag], x + threadIdx * tile_size - 1,
                     sizeof(float) * (tile_size + 1), MemcpyGlobalToSpm);
   }
   // traverse all tiles
   for (int i = threadIdx; i < tile_num; i += threadDim) {
        const int tile_size = (i < tile_num - 1) ? max_tile_size : last_tile_size;


        // wait async memcpy to finish
        memcpy_wait();
        // read next tile of x
        if (i + threadDim < tile_num) {
            const int next_i = i + threadDim;
            const int next_tile_size = (next_i < tile_num - 1) ? max_tile_size : last_tile_size;
            memcpy_async(tile_x[flag ^ 1], x + next_i * next_tile_size - 1,
                            sizeof(float) * (next_tile_size + 1), MemcpyGlobalToSpm);
        }


        // perform 3-point smoothing
        if (i == 0) {
            tile_x[flag][0] = 0.0f;
        }
        if (i == tile_num - 1) {
            tile_x[flag][tile_size] = 0.0f;
        }
        perform_3pt_smooth(tile_x[flag], tile_size, tile_y);


        // output tile of y
        memcpy(y + i * tile_size, tile_y, sizeof(float) * tile_size);


        // update
        flag ^= 1;
   }


   // free SPM
   free(tile_x[0]);
   free(tile_x[1]);
   free(tile_y);


    // over
    return;
}
```

**优化效果**

在 SPE 主频为 2.36GHz，输入输出数据皆为 float 类型，输入信号长度 N = 512k 的场景下，测试得到双缓冲版本的耗时只有基础版本的 78% 左右。双缓冲设计通过将输入信号的读取过程与卷积计算过程并行，从而实现加速。

#### 4.3.2 广播优化

**优化策略**

针对多个计算核心，需要同时读取 Global 存储中同一块数据的场景，可以利用广播（Broadcast）进行优化。

例如，在稠密矩阵与向量乘（GEMV）的计算中，可以将输入矩阵的不同行分配到不同计算核心上，然后分别与输入向量相乘，最终再将结果分别输出。其中，不同计算核心需要从 Global 存储中读取同一个输入向量。

此时，如果各个计算核心分别读取同样的 Global 存储空间，会产生明显的浪费。但广播机制，允许一个计算核心将 Global 存储上的数据读入自己的 SPM 存储器，然后通过广播同时传输给多个不同计算核心的 SPM 存储器。

合理利用广播机制，可以减少对 Global 存储器的访问次数，提升数据读取的效率。具体使用方法，可以参考 SDAA C 编程指南的 Broadcast 数据搬运（详情请参考《SDAA C 编程指南》的 Broadcast 数据搬运章节）章节。

**优化示例**

本节以一个稠密矩阵与向量乘积的算法为例，演示广播机制对 Global 存储上重复数据访问的性能优化。

**普通实现**

假设输入矩阵的行数等于 SPA 中所有 SPE 的总数，输入矩阵 A 的各行分给不同的 SPE，每个 SPE 每次只需要负责 A 的某一行与输入向量 x 的内积计算。其 Kernel 函数如下：

```c
__global__ void test_gemv_32xN_baseline(float* A, float* x, const int N, float* y) {
    // max size of one tile
    constexpr int max_tile_size = 16384;
    // allocate SPM
    float* tile_A = (float*)malloc(sizeof(float) * max_tile_size);
    float* tile_x = (float*)malloc(sizeof(float) * max_tile_size);
    // parallelize
    float* pA = A + threadIdx * N;
    float* px = x;


    // iterate
    int rest = N;
    float result = 0.0f;
    while (rest > 0) {
        const int tile_size = rest > max_tile_size ? max_tile_size : rest;
        // load tile of A into SPM
        memcpy(tile_A, pA, sizeof(float) * tile_size);
        // load tile of x into SPM


        memcpy(tile_x, px, sizeof(float) * tile_size);
        // compute dot product
        result += dev_dot_product(tile_A, tile_x, tile_size);
        // update
        rest -= tile_size;
        pA += tile_size;
        px += tile_size;
    }


    // output
    y[threadIdx] = result;
    // free SPM
    free(tile_A);
    free(tile_x);
    // over
    return;
}
```

其中 `dev_dot_product` 是负责向量内积计算的 device 函数（出于简便考虑，未处理边界情况）：

```c
__device__ float dev_dot_product(const float* x, const float* y, const int N) {
    const float *px = x, *py = y;
    floatv16 vres[4];
    floatv16 vx[4], vy[4];
    vres[0] = vres[1] = vres[2] = vres[3] = 0;
    for (int i = 0; i < N; i += 64) {
        simd_load(vx[0], px);
        simd_load(vy[0], py);
        vres[0] += vx[0] * vy[0];


        simd_load(vx[1], px + 16);
        simd_load(vy[1], py + 16);
        vres[1] += vx[1] * vy[1];


        simd_load(vx[2], px + 32);
        simd_load(vy[2], py + 32);
        vres[2] += vx[2] * vy[2];


        simd_load(vx[3], px + 48);
        simd_load(vy[3], py + 48);
        vres[3] += vx[3] * vy[3];


        px += 64;
        py += 64;
    }


    vres[0] += vres[2];
    vres[1] += vres[3];
    vres[0] += vres[1];


    float res[16];
    simd_store(vres[0], res);
    // stride = 8
    res[0] += res[0 + 8];
    res[1] += res[1 + 8];
    res[2] += res[2 + 8];
    res[3] += res[3 + 8];


    res[4] += res[4 + 8];
    res[5] += res[5 + 8];
    res[6] += res[6 + 8];
    res[7] += res[7 + 8];
    // stride = 4
    res[0] += res[0 + 4];
    res[1] += res[1 + 4];
    res[2] += res[2 + 4];
    res[3] += res[3 + 4];
    // stride = 2
    res[0] += res[0 + 2];
    res[1] += res[1 + 2];
    // stride = 1
    res[0] += res[0 + 1];


    // over
    return res[0];
}
```

**优化实现**

由于输入向量 x 是所有 SPE 计算过程中都要用到的，因此，可以采用广播机制避免对 Global 存储的重复访问，改善访存带宽。具体做法是先由一个 SPE（假定为 SPE0）从 Global 存储中读取 x，然后通过一次全阵列广播，将 x 分发给所有 SPE。相应的 Kernel 函数代码如下：

```c
__global__ void test_gemv_32xN_v1(float* A, float* x, const int N, float* y) {
    // max size of one tile
    constexpr int max_tile_size = 16384;
    // allocate SPM
    float* tile_A = (float*)malloc(sizeof(float) * max_tile_size);
    float* tile_x = (float*)malloc(sizeof(float) * max_tile_size);
    // parallelize
    float* pA = A + threadIdx * N;
    float* px = x;


    BroadcastHandle handle;
    ThreadGroup thread_group(0xFFFFFFFF);
    thread_group_exclude(thread_group, 0);
    broadcast_set_thread_group(handle, &thread_group);
    // iterate
    int rest = N;
    float result = 0.0f;
    while (rest > 0) {
        const int tile_size = rest > max_tile_size ? max_tile_size : rest;


        if (0 == threadIdx) {
              memcpy(tile_x, px, sizeof(float) * tile_size);
        }
        sync_threads();
        broadcast_async(tile_x, tile_x, sizeof(float) * tile_size, 0, BroadcastSpmToSpm, handle);


        // load tile of A into SPM
        memcpy(tile_A, pA, sizeof(float) * tile_size);


        broadcast_wait(handle);
        // compute dot product


        result += dev_dot_product(tile_A, tile_x, tile_size);
        // update
        rest -= tile_size;
        pA += tile_size;
        px += tile_size;
    }


    // output
    y[threadIdx] = result;
    // free SPM
    free(tile_A);
    free(tile_x);
    // over
    return;
}
```

**优化效果**

对于输入矩阵为 32×32768 的场景，在 SPE 主频 2.36GHz 的情况下，测试得到使用广播优化后的耗时可以降低到优化前 82% 左右。

## 5. 优化实战

### 5.1 优化思路

本章将以求解下三角线性方程组为例，演示如何综合运用此前各章节介绍的优化技术。

**求解下三角线性方程组**

求解线性方程组的一个常用方法是 LU 分解：将系数矩阵分解为下三角和上三角矩阵的乘积，然后依次求解上下三角线性方程组。

由于三角化后的线性方程组可以很容易地逐行消元求解，因此，这种方法的效率很高，适用于大多数线性方程组的求解。

算法设计阶段，通常可以先从串行版本开始，构建算法的整体逻辑，然后在此基础上进行并行改造，最后做进一步的优化。

**优化前：串行实现**

假设待求解的下三角线性方程组记为：Lx = b

其中，L 是 N 阶的下三角矩阵，x 和 b 均为 N 维列向量。

按照三角方程组的逐行消元法，串行版本伪代码如下：

```c
for (i = 0; i < N; i++) {
    // Initialize the sum to zero for each row i
    sum = 0;


    // Compute the dot product of the already solved parts with the corresponding coefficients in row i
    for (j = 0; j < i; j++) {
        sum += L[i][j] * x[j];
    }


    // Solve for x[i] using the computed sum and the known value b[i], then normalize by the diagonal
element L[i][i]
    x[i] = (b[i] - sum) / L[i][i];
}
```

串行版本需要逐行消元求解，各行相互依赖，无法并行，因此很难获得好的性能。

**基础优化：并行实现**

串行版本实际上就是利用已求解的部分，对尚未求解的方程进行消元。因此，每当求解出 x 的一个分量，就可以立即对所有剩下的方程消元，这样就可以将消元的过程并行化，并行版本伪代码如下：

```c
for (i = 0; i < N; i++) {
    // Solve for x[i] directly using the diagonal element L[i][i]
    x[i] = b[i] / L[i][i];


    // Update the remaining elements of b by subtracting the effect of the newly solved x[i]
    for (j = i + 1; j < N; j++) {   // parallelizable loop
        b[j] -= L[j][i] * x[i];
    }
}
```

其中，第二个循环是可以并行化的。后续示例将在太初 AI 加速卡上，以此为基础，展开逐步优化。

### 5.2 优化程序

本节将综合应用计算和访存方面的优化手段，对示例进行逐步分析与优化。

**优化点 1：初步并行化**

加速卡上各个 SPE 可以并行执行消元循环，同时由一个固定的 SPE（假设为 SPE0）负责求解消元后的第一个待求方程的解，伪代码如下：

```c
i_beg = 0;
while (i_beg < N) {
    if (0 == threadIdx) {
        // SPE0 (threadIdx == 0) solves the first equation in the current segment
        b[i_beg] /= L[i_beg][i_beg];
    }
    // Synchronize threads
    sync_threads();


    // Store the latest solved value
    latest_x = b[i_beg];


    // Parallel loop: each thread performs elimination on a subset of equations
    for (i = 1 + i_beg + threadIdx; i < N; i += threadDim) {
        b[i] -= L[i][i_beg] * latest_x;
    }
    i_beg++;
}
```

**优化点 2：改善吞吐量**

初步并行化版本在执行并行消元时，每个线程最多只能对一个待求解方程的一项消元。在太初 AI 加速卡上，至少有两点不足：

- 一项消元只需要一次标量乘加运算，无法充分利用 SPE 的向量乘加指令，造成浮点性能的浪费。
- 对一个待求解方程的一项消元只需要访问该项系数，因此，对 Global 存储的访问只能以少量多次的方式进行，无法发挥 Global 存储的访存带宽。

针对以上两点问题，可以考虑增加并行消元过程中每个线程的吞吐量：每个线程最好能同时对多个待求解方程的连续若干项进行消元，进行进一步优化。即每一轮需要多求解几个方程，得到若干个连续的解，然后再多线程并行从剩余的待求解方程中，同时消去这些解对应的系数。

假设每次求解连续 16 个方程的解，每个线程并行按照每次 16 个的速度从剩余待求解方程中消去这 16 个解对应的系数。伪代码如下（假设 N 是 16 的整数倍）：

```c
i_beg = 0;
while (i_beg < N) {
    if (0 == threadIdx) {
        // SPE0 loads a 16x16 block of L corresponding to the next 16 equations
        load lt[0:15][0:15] = L[i_beg:i_beg+15][i_beg:i_beg+15];


        // SPE0 solves the 16 equations in the block using matrix inversion
        solve x[0:15] = inv(lt) * b[i_beg:i_beg+15];
    }


    // Update the global array b with the newly solved values
    b[i_beg:i_beg+15] = x[0:15];


    // Synchronize threads
    sync_threads();


    // Load the solved values into all threads
    load x[0:15] = b[i_beg:i_beg+15];


    // Parallel loop: each thread performs elimination on a subset of remaining equations
    for (i = i_beg + 16 * (1 + threadIdx); i < N; i += 16 * threadDim) {
        // Load a 16x16 block of L corresponding to the current set of equations
        load lt[0:15][0:15] = L[i:i+15][i_beg:i_beg+15];


        // Load the corresponding portion of b for the current set of equations
        load lb[0:15] = b[i:i+15];


        // Perform vectorized elimination on the loaded block
        for (j = 0; j < 16; j++) {
             lb[i:i+15] -= lt[0:15][j] * x[j];   // vectorizable operation
        }


        // Store the updated values back to the global array b
        b[i:i+15] = lb[0:15];
    }
    i_beg += 16;
}
```

当前版本中，每个线程都需要对连续 16 个待求解方程的连续 16 项进行消元，解决了上述两个问题：

- 由于系数矩阵采用了列优先存储，可以直接向量加载同一个解的系数，然后进行一次向量乘加运算将这些系数消去。
- 由于每次需要消去 16×16 个系数，每个线程都可以一次性从 Global 存储中加载这 256 个系数，可以明显改善 Global 存储的访存带宽。

### 5.3 完整程序

**优化点 1：初步并行化**

初步并行化的 Kernel 函数实现如下：

```c
__global__ void test_solve_LT_v1(float* LT, float* b, const int N) {
    // assert: shape of LT = N x N, N is multiple of 16
    int i_beg = 0;


    while (i_beg < N) {
        // current equation to solve: LT(i_beg:N-1, i_beg:N-1)*x(i_beg:N-1) = b(i_beg:N-1);
        if (0 == threadIdx) {
              b[i_beg] /= LT[get_index(i_beg, i_beg, N)];
        }
        sync_threads();


        // update
        float latest_x = b[i_beg];


        // elimination process (parallelizable)
        for (int i = 1 + i_beg + threadIdx; i < N; i += threadDim) {
              b[i] -= LT[get_index(i, i_beg, N)] * latest_x;
        }


        i_beg++;
    }
    // over
    return;
}
```

其中，为了将矩阵元素的 2 元脚标映射为列主元存储的 1 维索引，设计了一个映射函数：

```c
__device__ inline int get_index(const int i, const int j, const int N) {
    return (((2 * N - i + 1) * i) >> 1) + j - i;
}
```

**优化点 2：改善吞吐量**

实现 Kernel 函数的过程中，可以在上节伪代码的基础上，新增了两项优化：

1. 在并行消元的过程中，由于所有线程都要使用新求得的 16 个解。因此，可以通过一次广播将其分享给所有线程，避免了对 Global 存储的重复访问。
2. 当 N 较大时，在一轮之内，每个线程必须进行多次消元（每次 16 个方程）。此时，可以采用双缓冲思想，在对当前 16 个方程进行消元的同时，预取接下来 16 个方程的系数。

其 Kernel 函数的代码实现如下：

```c
__global__ void test_solve_LT_v2(float* LT, float* b, const int N) {
    // assert: shape of LT = N x N, N is multiple of 16
    // allocate SPM


   float* local_lt = (float*)malloc(sizeof(float) * ((1+16)*16/2));


   int flag = 0;
   float* local_eq_coeff_mem = (float*)malloc(2 * (sizeof(float) * 16 * 16));
   float* local_eq_coeff[2];
   local_eq_coeff[0] = local_eq_coeff_mem;
   local_eq_coeff[1] = local_eq_coeff_mem + 16 * 16;


   float* local_b_x_mem = (float*)malloc(3 * sizeof(float) * 16);
   float* local_b[2];
   local_b[0] = local_b_x_mem;
   local_b[1] = local_b_x_mem + 16;
   float* local_x = local_b_x_mem + 32;


   // broadcast handle
   BroadcastHandle handle;


   // main loop
   int i_beg = 0;
   while (i_beg < N) {
       // current equation to solve: LT(i_beg:N-1, i_beg:N-1)*x(i_beg:N-1) = b(i_beg:N-1);
       if (0 == threadIdx) {
           // load LT(i_beg:i_beg+15, i_beg:i_beg+15) into local_lt
           // load b(i_beg:i_beg+15) into local_b
           {
                  auto dst = local_lt;
                  for (int j = 0; j < 16; j++) {
                      auto src = LT + get_index(i_beg + j, i_beg + j, N);
                      memcpy_async(dst, src, sizeof(float) * (16 - j), MemcpyGlobalToSpm);
                      // update
                      dst += (16 - j);
                  }


                  memcpy(local_b[0], b + i_beg, sizeof(float) * 16);
           }
           // wait for load to be finish
           memcpy_wait();


           // solve x(i_beg:i_beg+15) = inv(local_lt)*local_b
           solve_16x16_LT(local_lt, local_b[0]);


           // write back to update b(i_beg:i_beg+15)
           memcpy_async(b + i_beg, local_b[0], sizeof(float) * 16, MemcpySpmToGlobal);
       }
       sync_threads();


       // RMA broadcast local_b (from thread-0) to local_x (of all threads)
       broadcast_async(local_x, local_b[0], sizeof(float) * 16, 0, BroadcastSpmToSpm, handle);
       broadcast_wait(handle);
       // wait for write back to finish
       memcpy_wait();
       sync_threads();


       // eliminate local_x[] from rest equations
       // load coefficients of the 1st 16 equations
       // load rhs (b) of the corresponding equations
       if (i_beg + 16 * (1 + threadIdx) < N) {
           const int i = i_beg + 16 * (1 + threadIdx);
           auto dst = local_eq_coeff[flag];


              for (int j = 0; j < 16; j++) {
                  auto src = LT + get_index(i, i_beg + j, N);
                  memcpy_async(dst, src, sizeof(float) * 16, MemcpyGlobalToSpm);
                  // update
                  dst += 16;
              }


              memcpy_async(local_b[flag], b + i, sizeof(float) * 16, MemcpyGlobalToSpm);
        }


        // loop to cancel coefficients (parallelizable)
        for (int i = i_beg + 16 * (1 + threadIdx); i < N; i += 16 * threadDim) {
              // wait for previous memcpy to finish
              memcpy_wait();


              // load coefficients of the next 16 equations
              // load rhs (b) of the corresponding equations
              if (i + 16 * threadDim < N) {
                  const int i_next = i + 16 * threadDim;
                  auto dst = local_eq_coeff[flag ^ 1];
                  for (int j = 0; j < 16; j++) {
                      auto src = LT + get_index(i_next, i_beg + j, N);
                      memcpy_async(dst, src, sizeof(float) * 16, MemcpyGlobalToSpm);
                      // update
                      dst += 16;
                  }


                  memcpy_async(local_b[flag ^ 1], b + i_next, sizeof(float) * 16, MemcpyGlobalToSpm);
              }


              // eliminate local_x[] from equations LT(i:i+15, i_beg:N-1)*x = b
              eliminate_16x16(local_eq_coeff[flag], local_x, local_b[flag]);


              // write back to update b(i:i+15)
              memcpy(b + i, local_b[flag], sizeof(float) * 16);


              // switch
              flag ^= 1;
        }
        // update
        i_beg += 16;
    }


    // release
    free(local_lt);
    free(local_eq_coeff_mem);
    free(local_b_x_mem);


    // over
    return;
}
```

其中，求解当前下三角方程组前 16 个解的计算过程被封装为 device 函数实现：

```c
__device__ void solve_16x16_LT(float* lt, float* b) {
    auto diag_ptr = lt;
    for (int i_beg = 0; i_beg < 32; i_beg++) {


        // solve
        float x = -b[i_beg] / diag_ptr[0];
        b[i_beg] = -x;
        // eliminate
        diag_ptr++;
        for (int j = i_beg + 1; j < 32; j++) {
              b[j] += diag_ptr[0] * x;
              diag_ptr++;
        }
    }
    // over
    return;
}
```

同样，负责对连续 16 个待求解方程进行消元的计算过程也被封装为一个独立的 device 函数：

```c
__device__ void eliminate_16x16(float* lt_blk, float* x, float* b) {
    floatv16 vec_b[2];
    vec_b[0] = vec_b[1] = 0.0f;


    for (int j = 0; j < 16; j += 2) {
        // expand to vector
        floatv16 vec_x[2];
        vec_x[0] = -x[j];
        vec_x[1] = -x[j + 1];


        // load vector
        floatv16 vec_lt[2];
        simd_load(vec_lt[0], lt_blk + j * 16);
        simd_load(vec_lt[1], lt_blk + (j + 1) * 16);


        // FMA
        vec_b[0] += vec_lt[0] * vec_x[0];
        vec_b[1] += vec_lt[1] * vec_x[1];
    }


    floatv16 vres;
    simd_load(vres, b);
    vec_b[0] += vec_b[1];
    vres += vec_b[0];
    simd_store(vres, b);


    // over
    return;
}
```

这里为了充分利用 SPE 的浮点算力，使用了向量指令加速消元过程。同时，分配了两组向量寄存器，构成两组数据无依赖的计算流水线，手动进行循环展开。

**优化效果**

在 SPE 主频为 2.36GHz 的硬件上测试，对一个 2048 阶的下三角方程组进行求解时，最终优化版本相较于初步优化版本的加速比约为 15.6。
