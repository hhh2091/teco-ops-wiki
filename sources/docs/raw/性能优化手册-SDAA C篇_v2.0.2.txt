3. 性能度量
3.1 性能指标
本文主要介绍衡量程序性能的关键指标：执行周期和访存带宽。

 执行周期

在SDAA C中，使用执行周期作为衡量程序性能的关键指标。

程序性能公式如下：

      指令 执行周期    时间
 性能 = 程序 ⋅ 指令 ⋅ 执行周期 = 程序 ⋅ 执行周期 ⋅ 时钟周期
  1 ​         ​   ​   ​
                        1 ​




其中：

   时钟周期（单位：秒）：计算核心SPE基本的时间单位，指发生一个最小事件所需的时间，由计算核心SPE的时钟
   频率决定。

   时钟周期的公式： 时钟周期 = SPE 的时钟频率
                        1       ​




   执行周期（单位：拍）：程序运行所消耗的时钟周期个数（也称作拍数，一拍对应一个时钟周期）。

由于时钟周期是由计算核心SPE的时钟频率所决定的固定值，那么执行周期越短，程序的运行效率越高效，程序性能越
好。因此，执行周期是衡量程序性能的关键指标。


        说明：

          非访存类指令的执行周期通常是固定的。

          访存类指令的执行周期与数据量、访存方式以及时钟频率有关。



 访存带宽

对于以数据读写为主的程序，访存带宽（即IO带宽）是通用的性能指标。

访存带宽是计算核心SPE在单位时间内读取或写入内存的数据量，通常以每秒传输的比特数或字节数进行衡量。

 理论带宽

理论带宽的计算公式如下：

理论带宽（Byte/s） = 内存最大频率×内存通道数×648（每次传输位宽，单位：bit）    ​




 访存带宽

在实际使用中，由于算法效率、内存延迟等均会影响数据传输的效率，因此访存带宽通常低于理论带宽。

访存带宽的计算公式如下：

访存带宽（Byte/s） = 数据传输量（Byte）×频率（Hz ）
                   执行周期（拍）                             ​




由以上公式可以看出：执行周期越短，访存带宽越大。


3.2 性能采样
使用 perf_start 、 perf_stop 和 perf_print 函数可以对SDAA C程序进行性能采样，采集目标代码段
的执行周期。目前，性能采样工具只输出两个主要的性能数据，一个是 perf_start 、 perf_stop 之间的执
行周期（延时拍数），另一个是指令缓存脱靶次数（Instruction Cache Miss）。


      说明：

      使用性能采样函数前，需要手动包含 sdaa_perf.h 头文件。


   perf_start（详情请参考《SDAA C编程指南》的perf_start章节）：开始性能采样接口，通过此接口标识性能
   采样的起始位置。
   perf_stop（详情请参考《SDAA C编程指南》的perf_stop章节）：结束性能采样接口，通过此接口标识性能
   采样的结束位置。
   perf_print（详情请参考《SDAA C编程指南》的perf_print章节）：在终端界面输出采集到的性能数据（延时
   拍数和指令缓存脱靶次数）。

 性能采样示例

      说明：

      为准确测量性能，可以循环多次调用被测程序并计算执行周期的平均值，以减少缓存未命中（Cache Miss）
      所带来的性能偏差。


使用 perf_start 和 perf_stop 进行性能采样的示例如下：

  #include <sdaa_perf.h>
  using namespace sdaa;


  #define NUM 51200
  #define FSIZE (NUM * sizeof(float))


  __global__ void kernel_sig(float *a, uint32_t num)
  {
      if (threadIdx == 0){
          perf_start();
          for (uint32_t i = 0; i < num; i++) {
              a[i] = sigmoidf(a[i]);
          }
          perf_stop();
          perf_print();

      }
  }


  int main()
  {
      sdaaSetDevice(0);
      // 主机端申请数据空间
      float *host_data = (float *)malloc(FSIZE);
      for (int i = 0; i < NUM; i++) {
           host_data[i] = (float)i / NUM;
      }
      float *dev_input = NULL;
      // 申请Device端数据空间
      sdaaMalloc((void **)&dev_input, FSIZE);
      // 将主机端数据复制到Device端
      sdaaMemcpy(dev_input, host_data, FSIZE, sdaaMemcpyHostToDevice);
      // 核函数运行计算
      kernel_sig<<<1>>>(dev_input, NUM);
      sdaaDeviceSynchronize();
      // 将计算好的数据从Device端复制到主机端
      sdaaMemcpy(host_data, dev_input, FSIZE, sdaaMemcpyDeviceToHost);


      free(host_data);
      sdaaFree(dev_input);
      return 0;
  }

4.1 程序并行
4.1.1 任务级并行
本文主要介绍任务级并行的性能优化策略和示例。

 优化策略

4.1.2 线程级并行（SPMD）
本文主要介绍线程级并行（SPMD）的性能优化策略和示例。

 优化策略

太初AI加速卡目前提供多个计算核心SPE，每个SPE构成一个独立的线程。线程级并行（SPMD（详情请参考《SDAA C
编程指南》的SPMD章节））主要指：每个计算核心SPE并行完成不同数据的相同任务，从而发挥设备的并行处理能
力。




                                图1 线程级并行（SPMD）

如上图，使用SPMD优化前是所有线程操作同一个数据，优化后是每个线程操作不同的数据。因此，线程级并行可以大
幅度提高数据处理的性能。

计算51200个数据的sigmoid输出值。

  #include <sdaa_perf.h>
  using namespace sdaa;


  #define NUM 51200
  #define FSIZE (NUM * sizeof(float))
  #define SIMD_NUM 16


  __global__ void kernel_add(float *a, uint32_t num)
  {
      perf_start();
      floatv16 va;
      // 每次循环计算是跨步为 threadDim*16，为每个SPE分配16个数据去计算
      for (uint32_t i = 0; i < num; i += (threadDim * SIMD_NUM)) {
          // 用 i+threadIdx*SIMD_NUM
          // 作为坐标，每个SPE都往向量寄存器装入16个float数
          simd_load(va, (a + i + threadIdx * SIMD_NUM));
          // 每个SPE 都做 add的运算
          va = va + 10;
          // 将 每个SPE的向量寄存器里的16个float数据， 存回到对应地址
          simd_store(va, (a + i + threadIdx * SIMD_NUM));
      }
      perf_stop();
      perf_print();
  }

4.2.1 SPE同步接口
本文主要介绍使用线程组同步SPE接口的性能优化策略和示例。

 优化策略

使用线程组同步SPE接口，加入线程组的SPE在满足以下条件时，性能较好：

SPE的ID号除以8所得的商或者余数相同。

例如：使用包括4个SPE的线程组进行同步时，线程组A={0, 1, 2, 3}或B={0, 8, 16, 24}的性能要优于线程组C={0, 9, 18,
27}。


 优化示例

对使用线程组A={0, 1, 2, 3}、B={0, 8, 16, 24}和C={0, 9, 18, 27}同步SPE接口的执行周期进行测试并比较。

每个测试均调用1000次sync_threads（详情请参考《SDAA C编程指南》的sync_threads章节）并计算平均值以
消除初次调用时cache影响。

 优化策略

4.2.3 RMA数据搬运
4.2.3.1 SPE ID相关
本文主要介绍RMA数据搬运的性能优化策略和示例。

 优化策略

进行RMA数据搬运的两个SPE的ID号x和y，在满足以下条件时，性能较好：

令D(x, y) = abs(x/8 - y/8) + abs(x%8 - y%8)，则D(x, y)的值越小，性能越好。

例如：SPE0 和 SPE9 的RMA数据搬运性能要优于 SPE0 和 SPE31 的RMA数据搬运性能。

4.2.4 Broadcast数据搬运
4.2.4.1 SPE ID相关
本文主要介绍使用线程组进行Broadcast数据搬运的性能优化策略和示例。

 优化策略

使用线程组进行Broadcast数据搬运时，当线程组内的所有SPE满足以下任意一种条件时，性能较好：

    线程组内所有SPE刚好能被划分为：{0~7}，{8~15}，{16~23}，{24~31}中的一组或多组。

    线程组内所有SPE刚好能被划分为：{0, 8, 16, 24}，{1, 9, 17, 25}，{2, 10, 18, 26}，{3, 11, 19, 27}，{4, 12, 20,
    28}，{5, 13, 21, 29}，{6, 14, 22, 30}，{7, 15, 23, 31}中的一组或多组。

例如：对包含8个SPE的线程组进行广播时，线程组A={0, 1, 2, 3, 4, 5, 6, 7}或B={0, 1, 8, 9, 16, 17, 24, 25}的性能要优于
线程组C={0, 1, 9, 10, 18, 19, 27, 28}。

 优化示例

对使用线程组A={0, 1, 2, 3, 4, 5, 6, 7}、B={0, 1, 8, 9, 16, 17, 24, 25}和C={0, 1, 9, 10, 18, 19, 27, 28}进行阻塞型广播的
执行周期进行测试并比较。

每个测试均调用1000次broadcast（详情请参考《SDAA C编程指南》的broadcast 老接口章节）并计算平均值以消
除初次调用时cache影响。

4.2.6 矩阵乘
4.2.6.1 阻塞型矩阵乘
矩阵乘阻塞型接口相比非阻塞型接口具有更少的使用限制和更高的通用性，但相应地性能也会差于非阻塞型接口。当
满足一定对齐条件时，矩阵乘阻塞型接口的性能可以提升到接近非阻塞型接口的性能。

 优化策略

当阻塞型矩阵乘接口满足一下条件时，性能较好：

k（输入矩阵列数、权重矩阵行数）和n（权重矩阵列数）均是32的整数倍。

当使用阻塞型矩阵乘接口时，预先将矩阵分块为32大小的整数倍，可以有效提高矩阵乘接口的性能。

 优化示例

单一线程使用阻塞型矩阵乘接口计算SPM上矩阵的乘积时，测试以下两种情况的执行周期并比较：

   k和n均为32的整数倍（m=128，k=64，n=32）。

   k和n不是32的整数倍（m=128，k=60，n=30）。

本文主要介绍非阻塞型矩阵乘的性能优化策略和示例。

 优化策略

如果搬运的数据与当前程序的后续操作不存在数据依赖时，使用非阻塞式接口（详情请参考《SDAA C编程指南》的非

4.4.1 循环展开
本文主要介绍使用循环展开的方式提高性能的优化策略和示例。




文档版本：v2.0.2                           版权所有 © 太初(无锡)电子科技有限公司                 79
性能优化手册-SDAA C篇                                                  性能优化




 原理介绍

循环展开是在编译程序中，通过减少循环迭代次数进而提高程序执行速度的方法。

其核心思想是将操作对象线性化，在一次迭代中访问线性数据中的一组而非单个，使程序在执行更少的迭代次数后，
能够有效降地低循环的开销。同时，循环展开可以减少指令的相关性，增加循环体内语句并发执行的可能性，有助于
进行更好的指令调度；使用循环展开还可以减少分支预测失败的可能性。但循环展开也会使代码膨胀；代码的可读性
降低，以及寄存器压力增大。

循环展开可以在程序编写时完成，也可以由编译器自动完成。由于编译器可以更好地了解目标体系结构的工作原理，
并能够在有益的情况下展开循环。因此，建议您使用编译器自动展开，如果您手动展开，建议您在测试手动循环展开
确实有效再进行。

 优化策略

通过以下编译指示，控制是否需启用循环展开功能，以及展开的次数：

     #pragma clang loop unroll(enable) ：启用基于自动诊断的展开功能。

     #pragma clang loop unroll(disable) ：禁用基于自动诊断的展开功能。

     #pragma clang loop unroll(full) ：完全展开一个循环。

     #pragma clang loop unroll_count(N) ：显式指定展开N次，其中N是编译时的常量。

4.4.2 链接时优化（LTO）
本文主要介绍使用链接时优化（LTO）提高性能的优化策略和示例。

 原理介绍

链接时优化（LTO）是对整个SDAA C程序在链接时进行的跨模块间优化，聚焦于分析整个程序代码，属于过程间优化
（IPO）的一种。

具体实现原理是将一些pass的优化推迟到链接时再进行，从而使得这些pass具有可跨模块优化的能力。

LTO可以实现：

     更好的内联：LTO可以在链接阶段分析整个程序的调用关系，从而实现智能的函数内联决策，提高程序性能。



文档版本：v2.0.2                     版权所有 © 太初(无锡)电子科技有限公司          87
性能优化手册-SDAA C篇                                                                              性能优化




   消除死代码：LTO可以分析整个程序的数据流和控制流，检测并删除不会被执行的代码，从而减小程序大小。

   传播跨模块的常量：LTO可以在整个程序范围内传播已知的常量值，从而提高程序性能。

   消除跨模块的重复代码：LTO可以在整个程序范围内检测并消除重复的代码片段，从而减小程序大小。

 优化策略

通过以下编译选项，控制是否需启用LTO优化：

     -flto ：启用LTO优化，编译和链接时使用。

     --sdaa-link ：启用LTO优化，链接时使用，指示链接SDAA C的                            clang-offload-bundler 捆绑包。


 优化示例

分别使用以下两种方式编译示例程序，测试执行周期并比较：

   不使用LTO优化

   使用LTO优化

每个测试均调用10000次 foo1 函数并计算平均值以消除初次调用时cache影响。

 不使用LTO优化

执行以下编译命令，将 lib.scpp 编译成静态库，并使用 main.scpp 链接该静态库生成可执行文件：

  tecocc lib.scpp --sdaa-device-only -c -o nonlto.o
  # 将生成的.o打包成静态库
  ar -rcs nonlto.a nonlto.o
  # 编译main.scpp并链接静态库nonlto.a生成可执行文件
  tecocc main.scpp --sdaa-static-device-lib=nonlto.a -o nonlto.out

  # 将lib.scpp编译生成.o

4.4.3 浮点融合
本文主要介绍使用浮点融合提高性能的优化策略和示例。

 原理介绍

浮点融合是使用一条乘加/乘减运算操作代替乘法和加/减法两条运算的操作，从而提高程序性能并减小误差。

 优化策略

通过以下编译选项控制是否需启用浮点融合功能：
