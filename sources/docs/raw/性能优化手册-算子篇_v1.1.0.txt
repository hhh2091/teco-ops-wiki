3. 性能度量
 性能指标

本文对性能的定义主要从工程的角度出发，主要关注程序在处理器上运行的直接性能表现，即程序的执行速度，包含
两方面：

   以浮点运算为主的算法，采用浮点性能FLOPS（每秒执行的浮点运算数）作为关键指标。

   以数据读写为主的算法，采用IO带宽（通常以GB/s为单位）作为关键指标。

 计算方法

为了实际测算具体算法的性能指标，最常用的方法是测量算法在计算给定问题上的耗时，然后根据计算过程所需的浮
点运算数FLOPs或数据读写量来计算浮点性能与IO带宽。性能采样工具的使用，可以参考SDAA C编程指南的性能采样
（详情请参考《SDAA C编程指南》的性能采样章节）章节。

计算公式：

   浮点性能（FLOPS）= 浮点运算次数 / 运算时间（秒）

   IO带宽（GB/s）= 传输的数据量（字节） / 传输时间（秒）


     说明：

     针对设备端的性能优化，暂不考虑主机端执行、主机-设备之间数据传输的开销，只需测量设备端Kernel函数
     或其中关键代码段的执行用时。

4. 性能优化
4.1 概述
并行算法优化过程并不保证浮点性能与IO带宽两种指标同时接近硬件的峰值，这是由特定算法的计算密度决定的。针
对特定算法进行优化，可以根据算法的计算密度，选择相应的优化方式。

 计算密度

计算密度，也称为计算访存比，指的是一个算法在执行过程中，进行的浮点运算次数与读写数据量的比值，单位：
FLOPs/Byte。计算密度越大，说明算法越集中于浮点计算，而非数据读写。

例如，对于一个输入张量形状为NHWC，使用形状为3x3的2D卷积核，输出张量形状为NPQK的卷积运算，假设所有数
据类型均为float，其计算密度 I 为：

                                NPQK∗(2∗3∗3−1)∗C
                    I = (NPQK+N HW C+C∗3∗3∗K)∗ sizeof ( float )

 算法类别

对于特定硬件，峰值IO带宽与浮点性能的比值是确定的。因此，算法用于浮点运算和数据读写的时间之比完全取决于
计算密度。据此，可以将不同的算法进行分类：

   运算时间远大于数据读写时间，称为计算密集型算法。

   运算时间远小于数据读写时间，称为访存密集型算法。

   运算与数据读写时间相当，称为均衡型算法。

其中：

   对于密集型算法，其执行速度由对应行为的性能指标决定。因此对其展开优化，应针对相应的性能指标，使其尽
   量接近硬件的峰值性能。
   对于均衡型算法，其整体性能同时受计算和访存两方面影响，两方面的性能指标优化都能带来明显的收益。同
   时，如果能利用硬件的特性，设法让运算与数据读写过程并发执行，可以进一步降低算法的执行用时，提升整体
   性能。

后续章节将分别介绍计算和访存两方面的优化手段，并给出相应的优化实践案例。基于这些优化方法，用户可以结合
自己的需求，既可用于相应的密集型算法的优化，也可同时运用于均衡型算法的性能提升。

并行程序设计的首要原则是尽量将任务拆分成可以并行处理的子任务，分配给处理器的多个计算单元并行处理。

每个计算核心在执行运算时，应当充分利用现代向量处理器的向量计算能力和流水线设计，尽可能实现数据并行和指
令并行，才能发挥硬件的峰值性能。

此外，除了通用运算和控制功能外，某些处理器会提供针对特殊运算的硬件，在合适的场景下，能够获得明显优于通
用运算的性能表现。

4.2.1 向量指令使用
 优化策略

单指令多数据（SIMD）是一种典型的数据并行方法，指的是通过一条指令同时对多个不同的数据进行运算的设计思
想。在现代向量处理器中，SIMD就是由各种向量指令支持的。

例如，普通的一条标量加法指令只能完成一个加数与一个被加数的求和运算。而一条向量加法指令则可以一次完成多
个加数与等量的被加数的求和。虽然两种指令的延迟可能存在一定差异，但由于向量指令能一次完成更多的运算，其
整体运算速度是明显高于标量运算的。

因此，在算法设计以及性能调优的过程中，应当尽可能采用SIMD的思路，广泛使用向量指令进行加速。

需注意，SIMD中并行的数据，需要相互不存在数据依赖。因此，使用SIMD时，需要设法解除原本的依赖关系。在加法
示例中，对N个输入整数求和，顺序求和算法，意味着每次加法依赖于之前求和的结果，所有数据间产生了依赖关系，
因此无法进行数据并行。但是利用加法的结合律，将N个输入均分为若干组，每一组可以无依赖地分别求和，最终对各
组的结果求和得到输出。这样就实现了各组之间的数据并行，可以使用向量指令进行加速。

因此，在算法设计层面，需要开发者充分理解SIMD的思想，主动消除数据依赖关系，才能广泛地应用向量指令实现加
速。在太初AI加速卡上支持的向量指令的功能和使用方法可以参考SDAA C编程指南中的向量操作（详情请参考
《SDAA C编程指南》的向量操作章节）章节。

当 N 足够大时，离散积分的结果可以作为定积分精确值的高度近似。因此，将积分区间拆分到各个计算核心上，每
个核心负责一个子区间的积分，最终由一个核心（假设为SPE0）将各子区间的积分结果求和，即可输出最终的积分
值。伪代码实现如下：

   // Calculate the width of each subinterval
   float delta = (b - a) / N;


   // Determine the starting point of the subinterval for the current thread
   float x_beg = a + delta * threadIdx * N / threadDim;


   // Determine the ending point of the subinterval for the current thread
   float x_end = a + delta * (1 + threadIdx) * N / threadDim;


   // Compute the integral over the subinterval assigned to the current thread
   float sub_integral = integrate(x_beg, x_end, N / threadDim);


   // Synchronize all threads to ensure all sub_integrals are computed
   sync_threads();


   if (0 == threadIdx) {
       // If this is the first thread (threadIdx == 0),
       // gather sub_integral values from all other threads using RMA (Remote Memory Access)
       RMA get sub_integral from all other threads;


       // Sum up all the sub_integral values to get the total integral
       float sum = sum all sub_integral;


       // Write back the final result of the integration
       write back sum;
   }



其中，单个子区间的积分是通过函数 integrate() 的矩形公式实现的。

以 f (x) = x2 ，[a, b] = [−3, 3]，N = 106 为输入参数，使用上述并行策略，在SPE主频为2.36GHz的条件下，
使用SDAA C编程指南的性能采样功能（详情请参考《SDAA C编程指南》的性能采样章节）计算Kernel函数中并行积
分计算代码的执行用时（不包括RMA通讯用时），平均约为107us。

 优化实现

基于上述并行版本，可以进一步优化，即在负责每个计算核心的子区间积分过程中，进行数据并行：将子区间内所有
参与离散积分的点的积分过程拆分为16组，即：
                                           16            N /16
                                 I = ∑i=1 ∑j=1 f (xi + 16(j − 1)△)△

这16组积分过程是独立的，只需要最终将分别积分的结果累加即可。这样就构造了16组不存在数据依赖的计算过程，可
以用太初AI加速卡的单精度浮点数向量指令来同时运算。因此， integrate() 函数的实现如下：

   __device__ float integrate_v1(const float a, const float b, const int N) {
       // discrete interval
       const float dx = (b - a) / N;
       // compute integral, using 0-th order approximation
       float integral = 0.0f;
       floatv16 vx = a;
       floatv16 vdx = dx;
       {
           floatv16 offset =
                 {0.0f, 1.0f, 2.0f, 3.0f, 4.0f, 5.0f, 6.0f, 7.0f, 8.0f, 9.0f, 10.0f, 11.0f, 12.0f, 13.0f, 14.0f,
           vx += offset * vdx;
       }
       floatv16 vres = 0.0f;


       // loop and use SIMD
       int i;
       for (i = 0; i <= N - 16; i += 16) {
           vres += vx * vx;
           vx += vdx;
       }


       // loop the rest part
       float x = a + i * dx;
       while (i < N) {
           integral += x * x;
           x += dx;
           i++;
       }


       // reduce result
       float res[16];
       simd_store(vres, res);
       for (i = 0; i < 16; i++) {
           integral += res[i];
       }
       integral *= dx;
       // over
       return integral;
   }




 优化效果

采用上述向量指令优化后，在相同条件下测得的Kernel执行用时，降低到平均8us，也就是获得了约13倍的加速比。


4.2.2 指令流水线优化
 优化策略

处理器的每个硬件核心通常有多个指令流水线，是实现多指令（MI）并行设计思想的基础。

所谓指令流水线，是将一条指令在处理器上执行的过程，分解为多个串行的阶段，而非一个不可拆分的整体，指令
（及其数据）随着时钟信号依次通过所有阶段。通过流水线化，处理器可以并行处理多个指令，仅轻微增加每条指令
的延迟。

指令流水线通过增加指令执行的阶段划分，提升了处理器的指令吞吐量（单位时间内能执行的指令条数）。此外，现
代处理器广泛采用多发射流水线技术：在流水线的每个阶段，可以同时处理不止一条指令，这同样可以大幅度提升处
理器吞吐量。

因此，为了充分利用处理器流水线提供的指令吞吐量，开发者应当在同一时间提供足够数量的相同指令（在寄存器等
资源足够的前提下），由处理器并行地执行这些指令，将指令流水线的各个阶段尽量占满。

 优化示例

在上节使用向量指令优化后的算法中，计算子区间积分的循环每次只有一条浮点向量乘加指令在执行。而太初AI加速卡
的每个计算核心都支持多级指令流水线，当多条无依赖的指令同时执行时，流水线才能被这些指令填满，达到最佳运
算性能。因此，可以尝试进一步的优化：在循环内同时执行多条浮点向量乘加指令，也就是循环展开。示例选择将循
环展开两次，即每轮循环内同时执行2条乘加指令，相应的Kernel代码如下所示：

      float integral = 0.0f;
      floatv16 vx[2], vres[2], vdx = dx;

4.2.3 矩阵乘法加速单元
 优化策略

不同处理器硬件由于面向的应用场景不同，往往会提供一些针对特定问题（例如特殊函数、矩阵乘法等）加速的计算

       matmul_store(handle, output, 128, MatmulN32);


       free(left_mat);
       free(right_mat);
       free(output);

       return;

4.3 访存相关优化
太初AI加速卡设备端的存储器模型可以分为3层：Global存储、SPM存储和寄存器。


文档版本：v1.1.0                            版权所有 © 太初(无锡)电子科技有限公司                            10
性能优化手册-算子篇                                               性能优化




   Global存储是所有片上所有核心共享的大容量存储器。

   SPM存储是每个计算核心的本地高速缓存，属于计算核心的私有资源。

   寄存器是核心执行运算和控制时，使用的特殊存储资源。

从访问速度而言，寄存器>SPM存储>Global存储。从存储容量而言，Global存储>SPM存储>寄存器。

而一个算法运行过程的数据流动通常包括如下过程：

  1. 从主机端将输入数据传输到Global存储。

 2. 计算核心从Global存储读取数据到自己的SPM中。

 3. 计算核心使用SPM中的数据进行计算，这个过程会根据指令执行的需要，使用各种寄存器，各种中间结果和阶段
   性输出都暂存在SPM中。
 4. 计算完成后，SPM上的输出结果写回Global存储。

 5. 主机端从Global存储中取回输出结果。




                             图1 数据流向示意图

其中，

   阶段1和5，主机端与Global存储之间的数据交换可以参考SDAA C编程指南中的数据传输（详情请参考《SDAA C
   编程指南》的数据传输章节）章节进行调用和优化。
   阶段3，计算过程中访问SPM以及使用寄存器的过程，通常不需要特殊的控制。

   阶段2和4，由于Global存储访问速度最慢，容量最大，输入输出阶段都需要对其进行大量的数据读写，因此是需
   要集中优化的阶段。

Global存储的具有单次访问延迟很高，但吞吐量（即单次访问的数据量）很大的特点。因此，它不适合像SPM或寄存
器一样，以高频多次少量的方式进行读写。Global存储最佳的读写模式是一次读写大量数据，控制访问频次。因此，
可以考虑尽可能实现：

   合并读取：将多次计算所需的输入数据作为一个整体，通过一次对Global存储的访问操作，读入SPM。

   合并写出：使用SPM上的数据进行计算，输出结果暂存在SPM上，将SPM上的结果整体写回Global存储。


4.3.1 双缓冲优化
 优化策略

主机与设备间的数据传输可以与设备上Kernel的执行并行，互不影响。这意味着：主机端可以通过异步的数据传输接
口发起传输，然后立即发起Kernel的执行操作，实现访存和计算的并行，进一步缩短程序耗时。同样的，使用Global
存储与SPM间的非阻塞型DMA（详情请参考《SDAA C编程指南》的非阻塞型DMA数据搬运章节）接口，可以实现访
存和计算的并行。

这意味着开发者需要解除并行数据传输与计算过程的数据依赖，也就是计算过程不能依赖于同一时刻正在传输的数
据。为了实现这一点，开发者需要在SPM上设置至少2个数据缓冲区。

   计算缓冲区：存储计算过程依赖的数据。

   传输缓冲区：存储异步传输的数据。

当计算和访存都完成后，再交换两个缓冲区的职能，就能始终确保计算过程与数据传输过程无依赖。

正是因为这种同时需要2个数据缓冲区的特点，此优化方法被称为双缓冲优化。

 优化示例

本节以1D卷积算法为例，演示如何使用双缓冲技术优化Global存储的数据访问。即输入一个长度为N的一维信号（单
通道），使用一个长度为3的卷积核对其进行滤波，输出长度仍为N的滤波结果。

 普通实现

算法的基本设计思路：

  1. 首先，需要根据SPM大小，确定每次最多能计算输出信号的长度dN。据此将计算过程分解为若干部分，由不同的
   SPE并行。

 2. 在每个SPE上，给定需要计算的输出信号范围后，先从Global存储中将计算过程所依赖的输入信号读入SPM，然
   后进行计算，并将结果从SPM的输出缓冲区写回Global存储的对应位置。

以伪代码描述各个SPE的并行计算过程如下：

   for (i = threadIdx; i < N / dN; i += threadDim) {

         Determine the input signal range needed for computation: x[i*dN-1, (i+1)*dN]

   SPM
         Perform the convolution filtering on spm_x and store the result in the output buffer spm_y in SPM
         Write the spm_y output buffer back to Global storage
   }

4.3.2 广播优化
 优化策略

针对多个计算核心，需要同时读取Global存储中同一块数据的场景，可以利用广播（Broadcast）进行优化。

例如，在稠密矩阵与向量乘（GEMV）的计算中，可以将输入矩阵的不同行分配到不同计算核心上，然后分别与输入向
量相乘，最终再将结果分别输出。其中，不同计算核心需要从Global存储中读取同一个输入向量。

此时，如果各个计算核心分别读取同样的Global存储空间，会产生明显的浪费。但广播机制，允许一个计算核心将
Global存储上的数据读入自己的SPM存储器，然后通过广播同时传输给多个不同计算核心的SPM存储器。

合理利用广播机制，可以减少对Global存储器的访问次数，提升数据读取的效率。具体使用方法，可以参考SDAA C
编程指南的Broadcast数据搬运（详情请参考《SDAA C编程指南》的Broadcast数据搬运章节）章节。

5. 优化实战
5.1 优化思路
本章将以求解下三角线性方程组为例，演示如何综合运用此前各章节介绍的优化技术。

 求解下三角线性方程组

求解线性方程组的一个常用方法是LU分解：将系数矩阵分解为下三角和上三角矩阵的乘积，然后依次求解上下三角线
性方程组。

由于三角化后的线性方程组可以很容易地逐行消元求解，因此，这种方法的效率很高，适用于大多数线性方程组的求
解。

算法设计阶段，通常可以先从串行版本开始，构建算法的整体逻辑，然后在此基础上进行并行改造，最后做进一步的
优化。

 优化前：串行实现

假设待求解的下三角线性方程组记为： Lx = b

其中， L 是 N 阶的下三角矩阵， x 和 b 均为 N 维列向量。

按照三角方程组的逐行消元法，串行版本伪代码如下：

   for (i = 0; i < N; i++) {
       // Initialize the sum to zero for each row i
       sum = 0;


       // Compute the dot product of the already solved parts with the corresponding coefficients in row i
       for (j = 0; j < i; j++) {
           sum += L[i][j] * x[j];
       }


       // Solve for x[i] using the computed sum and the known value b[i], then normalize by the diagonal
   element L[i][i]
       x[i] = (b[i] - sum) / L[i][i];
   }



串行版本需要逐行消元求解，各行相互依赖，无法并行，因此很难获得好的性能。

 基础优化：并行实现

串行版本实际上就是利用已求解的部分，对尚未求解的方程进行消元。因此，每当求解出 x 的一个分量，就可以立即
对所有剩下的方程消元，这样就可以将消元的过程并行化，并行版本伪代码如下：

   for (i = 0; i < N; i++) {
       // Solve for x[i] directly using the diagonal element L[i][i]
       x[i] = b[i] / L[i][i];


       // Update the remaining elements of b by subtracting the effect of the newly solved x[i]
       for (j = i + 1; j < N; j++) {   // parallelizable loop
           b[j] -= L[j][i] * x[i];
       }
   }



其中，第二个循环是可以并行化的。后续示例将在太初AI加速卡上，以此为基础，展开逐步优化。


5.2 优化程序
本节将综合应用计算和访存方面的优化手段，对示例进行逐步分析与优化。

 优化点1：初步并行化

加速卡上各个SPE可以并行执行消元循环，同时由一个固定的SPE（假设为SPE0）负责求解消元后的第一个待求方程的
解，伪代码如下：

   i_beg = 0;
   while (i_beg < N) {
       if (0 == threadIdx) {
           // SPE0 (threadIdx == 0) solves the first equation in the current segment
           b[i_beg] /= L[i_beg][i_beg];
       }
       // Synchronize threads
       sync_threads();


       // Store the latest solved value
       latest_x = b[i_beg];


       // Parallel loop: each thread performs elimination on a subset of equations
       for (i = 1 + i_beg + threadIdx; i < N; i += threadDim) {
           b[i] -= L[i][i_beg] * latest_x;
       }
       i_beg++;
   }




 优化点2：改善吞吐量

初步并行化版本在执行并行消元时，每个线程最多只能对一个待求解方程的一项消元。在太初AI加速卡上，至少有两点
不足：

   一项消元只需要一次标量乘加运算，无法充分利用SPE的向量乘加指令，造成浮点性能的浪费。

   对一个待求解方程的一项消元只需要访问该项系数，因此，对Global存储的访问只能以少量多次的方式进行，无
   法发挥Global存储的访存带宽。

针对以上两点问题，可以考虑增加并行消元过程中每个线程的吞吐量：每个线程最好能同时对多个待求解方程的连续
若干项进行消元，进行进一步优化。即每一轮需要多求解几个方程，得到若干个连续的解，然后再多线程并行从剩余
的待求解方程中，同时消去这些解对应的系数。

假设每次求解连续16个方程的解，每个线程并行按照每次16个的速度从剩余待求解方程中消去这16个解对应的系数。伪
代码如下（假设N是16的整数倍）：

   i_beg = 0;
   while (i_beg < N) {
       if (0 == threadIdx) {
           // SPE0 loads a 16x16 block of L corresponding to the next 16 equations
           load lt[0:15][0:15] = L[i_beg:i_beg+15][i_beg:i_beg+15];


           // SPE0 solves the 16 equations in the block using matrix inversion
           solve x[0:15] = inv(lt) * b[i_beg:i_beg+15];
       }


       // Update the global array b with the newly solved values
       b[i_beg:i_beg+15] = x[0:15];


       // Synchronize threads
       sync_threads();


       // Load the solved values into all threads
       load x[0:15] = b[i_beg:i_beg+15];


       // Parallel loop: each thread performs elimination on a subset of remaining equations
       for (i = i_beg + 16 * (1 + threadIdx); i < N; i += 16 * threadDim) {
           // Load a 16x16 block of L corresponding to the current set of equations
           load lt[0:15][0:15] = L[i:i+15][i_beg:i_beg+15];


           // Load the corresponding portion of b for the current set of equations
           load lb[0:15] = b[i:i+15];


           // Perform vectorized elimination on the loaded block
           for (j = 0; j < 16; j++) {
                lb[i:i+15] -= lt[0:15][j] * x[j];   // vectorizable operation
           }


           // Store the updated values back to the global array b
           b[i:i+15] = lb[0:15];
       }
       i_beg += 16;
   }



当前版本中，每个线程都需要对连续16个待求解方程的连续16项进行消元，解决了上述两个问题：

   由于系数矩阵采用了列优先存储，可以直接向量加载同一个解的系数，然后进行一次向量乘加运算将这些系数消
   去。
   由于每次需要消去16×16个系数，每个线程都可以一次性从Global存储中加载这256个系数，可以明显改善
   Global存储的访存带宽。
